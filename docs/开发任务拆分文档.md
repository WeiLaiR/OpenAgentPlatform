# Open Agent Platform (Java) 开发任务拆分文档

## 1. 文档目标

本文档用于指导项目从“设计阶段”进入“实现阶段”。

目标：

- 明确推荐开发顺序
- 控制前期复杂度
- 保证每个阶段都有可见成果
- 避免一开始同时推进过多模块导致混乱

原则：

- 先跑通主链路，再补增强能力
- 先后端闭环，再做前端体验
- 先单点可用，再做平台化能力
- 先实现，再优化

------

## 2. 总体开发阶段

建议拆成 7 个阶段：

1. 项目基础骨架
2. 基础聊天闭环
3. 会话与消息持久化
4. RAG 能力接入
5. MCP 管理与 Tool 接入
6. Agent + RAG 联合模式
7. Trace、Memory 与平台优化

------

## 3. 阶段拆分

------

## 阶段一：项目基础骨架

### 目标

完成前后端基础工程整理，确保项目可以稳定启动和联调。

### 后端任务

- 整理 Spring Boot 项目包结构
- 配置基础依赖
- 接入 MySQL
- 建立基础配置文件结构
- 建立统一响应体
- 建立全局异常处理
- 建立基础日志输出

### 前端任务

- 整理 Vue3 项目目录
- 接入路由
- 接入状态管理
- 接入 UI 组件库
- 建立基础页面骨架
- 封装基础请求工具

### 阶段产出

- 前后端项目均可启动
- 前端可访问后端健康检查接口
- 后端具备基础工程结构

### 暂不做

- 不急着接 LangChain4j
- 不急着做复杂页面
- 不急着做权限和登录

------

## 阶段二：基础聊天闭环

### 目标

先完成最小 AI 对话能力，跑通“用户输入 → 模型响应 → 前端展示”。

### 后端任务

- 接入 LangChain4j ChatModel
- 接入 StreamingChatModel
- 实现 `POST /api/v1/chat/send` 创建请求接口
- 实现 `GET /api/v1/chat/stream/{requestId}` SSE 接口
- 定义 `ChatSendRequest`、`ChatAnswerVO`
- 建立 requestId 贯穿机制

### 前端任务

- 实现聊天页面基础布局
- 支持输入消息
- 支持接收 SSE 流
- 支持流式展示回答
- 支持展示错误状态

### 阶段产出

- 用户可以在前端发消息
- 后端可调用本地 LLM
- 前端可看到流式回答

### 暂不做

- 不做 RAG
- 不做 MCP
- 不做复杂 trace 侧栏
- 不做多轮 memory

------

## 阶段三：会话与消息持久化

### 目标

让聊天从“临时调用”变成“有状态会话”。

### 后端任务

- 建立 `conversation`
- 建立 `conversation_message`
- 实现创建会话接口
- 实现查询会话列表接口
- 实现查询历史消息接口
- 聊天完成后保存 user/assistant 消息
- 统一会话模式字段

### 前端任务

- 左侧增加会话列表
- 支持创建会话
- 支持切换会话
- 支持加载历史消息

### 阶段产出

- 具备基础会话系统
- 页面刷新后聊天记录不丢
- 可以查看历史对话

### 暂不做

- 不做复杂消息编辑
- 不做消息重试/撤回
- 不做 memory 持久化

------

## 阶段四：RAG 能力接入

### 目标

完成知识库上传、解析、向量化、检索，并在聊天中支持启用 RAG。

### 后端任务

- 建立 `knowledge_base`
- 建立 `knowledge_file`
- 建立 `knowledge_segment`
- 接入 Apache Tika
- 实现文件上传接口
- 实现文件索引接口
- 接入 embedding 模型
- 接入 Milvus
- 完成知识库检索接口
- 聊天接口支持 `enableRag`

### 前端任务

- 实现知识库管理页面
- 支持创建知识库
- 支持上传文件
- 支持查看文件状态
- 支持勾选多个知识库
- 在聊天页增加 RAG 开关

### 阶段产出

- 能上传 pdf/docx/md/txt
- 能完成向量入库
- 开启 RAG 后能基于知识库回答
- 支持多知识库多选检索

### 暂不做

- 不做复杂 rerank
- 不做文档预览高级交互
- 不做自动知识库推荐

------

## 阶段五：MCP 管理与 Tool 接入

### 目标

完成 MCP server 热插拔与工具快照管理。

### 后端任务

- 建立 `mcp_server`
- 建立 `mcp_tool_snapshot`
- 接入 LangChain4j MCP
- 实现 MCP server 创建接口
- 实现连接测试接口
- 实现工具同步接口
- 同步时生成 `runtimeToolName`，保证跨 server 全局唯一
- 实现工具启停接口
- 聊天接口支持 `enableAgent`
- 按会话动态装配可用 tools

### 前端任务

- 实现 MCP 管理页面
- 支持新增 MCP server
- 支持启用/禁用 server
- 支持同步工具列表
- 支持查看工具 schema
- 聊天页增加 Agent 开关
- 支持会话绑定 MCP server

### 阶段产出

- 可以通过页面添加/管理 MCP
- 聊天时模型可调用 MCP tool
- 不需要改代码就能新增外部工具能力

### 暂不做

- 不做过多权限控制
- 不做复杂审批工作流
- 但保留高风险工具确认弹窗 / 人工确认开关
- 不做复杂工具市场

------

## 阶段六：Agent + RAG 联合模式

### 目标

完成你最核心的目标：RAG 和 Agent 可独立启用，也可联合工作。

### 后端任务

- 实现 ChatOrchestrator
- 按开关编排四种模式：
  - 纯聊天
  - RAG
  - Agent
  - RAG + Agent
- 检索结果注入模型上下文
- Tool 装配与调用链统一纳入主流程
- 输出是否使用了 RAG、是否调用了工具

### 前端任务

- 聊天页支持双开关
- 展示本轮是否触发 RAG
- 展示本轮是否触发 Tool Call
- 展示知识库命中片段摘要
- 展示工具调用摘要

### 阶段产出

- 平台主链路完成
- 你的核心愿景首次完整落地
- 项目从“聊天工具”升级成“AI Agent Platform”

### 暂不做

- 不追求特别复杂的 agent planner
- 不接多 agent 编排
- 不引入实验性过强的高级 agentic 流程

------

## 阶段七：Trace、Memory 与平台优化

### 目标

让系统更可观测、更稳定、更像平台。

### 后端任务

- 建立 `trace_event`
- 建立 `chat_memory_session`
- 建立 `chat_memory_message`
- 接入 LangChain4j ChatMemory
- 自定义 ChatMemoryStore(MySQL)
- 明确请求级覆盖与会话级配置优先级
- conversation 维度串行执行，避免并发破坏 memory
- 优先接入 LangChain4j 原生 observability / listener 事件
- 记录 RAG / Tool / Streaming / Error 事件
- 支持单独查询 trace
- 支持清空 memory

### 前端任务

- 实现 Trace 侧栏/时间线
- 展示 rag、tool_call、tool_result、answer_end 等事件
- 支持查看单轮链路
- 支持查看命中文档片段
- 支持查看工具出入参摘要

### 阶段产出

- 平台具备调试能力
- 多轮对话更稳定
- 前端有明显的平台感和可观测性

### 暂不做

- 不做 LangSmith 级别复杂 DAG 可视化
- 不做长期记忆高级策略
- 不做评测平台

------

## 4. 推荐开发顺序

建议按下面顺序推进，不要乱序：

1. 项目基础骨架
2. 基础聊天闭环
3. 会话与消息持久化
4. RAG
5. MCP
6. Agent + RAG 联合
7. Trace + Memory

### 为什么这样排

因为这条顺序能保证：

- 每一阶段都能看到成果
- 不会一开始同时碰模型、向量库、MCP、前端复杂交互
- 先把“能聊”做出来，再把“能查”做出来，再把“能动”做出来

------

## 5. 每阶段验收标准

为了避免做着做着失控，每阶段只看最核心的验收结果。

### 阶段一验收

- 前后端都能启动
- 能打通一个后端测试接口

### 阶段二验收

- 页面可正常发消息
- 模型可返回流式结果

### 阶段三验收

- 会话可创建、切换、查看历史

### 阶段四验收

- 文件可上传、索引、检索
- 聊天可结合知识库回答

### 阶段五验收

- MCP 可配置
- 模型可触发工具调用

### 阶段六验收

- RAG 与 Agent 可独立/联合启用

### 阶段七验收

- 能查看 trace
- memory 生效
- 多轮体验稳定

------

## 6. 当前建议的起步重点

如果你现在准备正式开工，我建议你先只盯住前三个阶段：

- 阶段一：项目基础骨架
- 阶段二：基础聊天闭环
- 阶段三：会话与消息持久化

因为只要这三步做完，你就已经拥有：

- 一个能跑的前后端项目
- 一个能聊天的 AI 系统
- 一个有会话与历史的产品雏形

这时候你再接 RAG 和 MCP，会非常顺。

------

## 7. 不建议现在立刻做的事情

这些事情先忍住，不然很容易分散注意力：

- 登录权限体系
- 多用户租户隔离
- Docker 一键部署
- CI/CD
- 复杂 prompt 管理后台
- 高级 Agent planner
- 多 agent 协同
- 评测平台
- 监控告警平台
- 非常复杂的前端视觉优化

------

## 8. 一句话总结

当前最正确的路线是：

**先做“能聊”，再做“能查”，再做“能调工具”，最后做“可观测和可配置的平台能力”。**
