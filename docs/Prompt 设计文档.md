# Open Agent Platform (Java) Prompt 设计文档

## 1. 文档目标

本文档用于定义项目中的 Prompt 体系，指导后续聊天、RAG、Agent、MCP Tool 调用等能力的提示词设计。

目标：

- 建立清晰的 Prompt 分层
- 避免不同场景共用一个“大杂烩 Prompt”
- 让后续调优有明确入口
- 先满足第一版可用，再逐步优化

原则：

- Prompt 按职责拆分
- Prompt 尽量模板化
- Prompt 先稳定，再微调
- 优先保证行为可控，而不是一开始追求“特别聪明”

------

## 2. Prompt 体系总览

建议将项目中的 Prompt 分成 5 类：

1. 全局系统 Prompt
2. 普通聊天 Prompt
3. RAG 场景 Prompt
4. Agent / Tool 调用 Prompt
5. 输出格式约束 Prompt

它们不是并列覆盖，而是按场景进行组合。

------

## 3. Prompt 设计原则

------

## 3.1 不要所有场景共用一个 Prompt

不要把以下内容全部堆在一个系统提示词里：

- 你是 AI 助手
- 你要结合知识库
- 你可以调用工具
- 你必须输出 JSON
- 你必须简洁
- 你必须展示引用
- 你必须先思考再回答

这样做的结果通常是：

- 维护困难
- 调试困难
- 不同模式之间互相干扰

更好的方式是：

- 全局定义角色
- 按场景注入额外 Prompt
- 输出格式单独控制

------

## 3.2 先控制行为，再追求效果

第一版 Prompt 设计最重要的是：

- 不乱答
- 不胡编知识库内容
- 不乱调工具
- 结果结构尽量稳定

先做到“可控”，再去调“更聪明”。

------

## 3.3 Prompt 尽量参数化

不要把所有业务信息直接写死在 prompt 里。

例如不要写：

- 知识库里有用户手册、数据库设计、开发文档

而应该通过变量注入：

- 当前选中的知识库名称
- 当前可用工具列表
- 当前模式开关

------

## 4. 第一版需要的 Prompt 列表

建议第一版至少准备下面这些：

### 4.1 `system-base`

全局系统 Prompt

### 4.2 `chat-base`

普通聊天 Prompt

### 4.3 `rag-context`

RAG 场景 Prompt

### 4.4 `agent-tool-policy`

Tool 调用约束 Prompt

### 4.5 `answer-format`

最终回答格式约束 Prompt

### 4.6 `mcp-tool-selection`

MCP 工具使用提示 Prompt

你不一定非要把它们都落成数据库模板，第一版先做成代码中的模板也可以。

------

## 5. 各 Prompt 职责设计

------

## 5.1 全局系统 Prompt

### 作用

定义模型的基础角色、基本行为边界、回答风格。

### 解决的问题

- 你是谁
- 你应该怎么回答
- 哪些事情不能做
- 遇到不确定情况怎么办

### 第一版建议包含的内容

- 你是一个面向开发者的 AI 平台助手
- 回答要基于当前上下文
- 如果上下文不足，要明确说明不确定
- 不要编造不存在的事实
- 如果提供了知识库片段，优先基于知识库回答
- 如果有工具可用，只在必要时调用
- 最终回答尽量清晰、直接、结构化

### 第一版示例

```
你是一个面向开发与知识问答场景的 AI 助手。

你的目标是根据用户问题、对话上下文、知识库检索结果和可用工具，给出准确、清晰、可执行的回答。

请遵循以下原则：
1. 优先依据当前提供的上下文回答，不要编造不存在的信息。
2. 如果上下文不足以支持确定结论，请明确说明不确定。
3. 如果知识库内容已提供，应优先依据知识库内容作答。
4. 只有在确实需要外部能力时才调用工具，不要为了调用而调用。
5. 回答应清晰、简洁、有条理。
```

------

## 5.2 普通聊天 Prompt

### 作用

用于纯聊天模式下的基础行为约束。

### 解决的问题

- 纯 LLM 模式应该怎么答
- 不启用 RAG / Agent 时如何保持回答质量

### 第一版建议

这个 Prompt 可以很轻，不要太复杂。

### 第一版示例

```
当前未启用知识库检索和外部工具。
请仅根据用户问题和当前对话上下文进行回答。
如果你无法确定答案，请直接说明你的不确定性，不要虚构细节。
```

------

## 5.3 RAG 场景 Prompt

### 作用

告诉模型如何使用知识库片段。

### 解决的问题

- 检索结果该怎么用
- 当知识库命中与常识冲突时如何处理
- 当知识库不足时如何表述

### 第一版建议包含

- 下方提供的是知识库检索结果
- 应优先使用这些内容回答
- 不要把未检索到的信息说成来自知识库
- 如果检索结果不足，可以说明“根据当前检索结果无法确认”

### 第一版示例

```
下面提供的是与当前问题相关的知识库检索结果。

请优先依据这些检索结果回答问题。
要求：
1. 不要将未出现在检索结果中的内容说成知识库结论。
2. 如果检索结果可以支持回答，请尽量基于检索结果组织答案。
3. 如果检索结果不足以回答，请明确说明“根据当前知识库检索结果无法确认”。
4. 如果问题与检索结果无明显相关性，可以结合一般能力补充说明，但要区分哪些内容来自检索结果，哪些不是。
```

### 注入方式建议

后端在 prompt 中注入：

- 当前知识库列表
- 检索结果片段
- 每个片段的来源文件名

------

## 5.4 Agent / Tool 调用约束 Prompt

### 作用

告诉模型在什么情况下调用工具，以及调用时遵守什么规则。

### 解决的问题

- 避免乱调工具
- 避免无意义调用
- 避免把工具调用当作默认行为

### 第一版建议包含

- 工具是可选的，不是必须的
- 只有用户问题需要外部信息或执行能力时才调用
- 如果上下文已经足够，不要调用工具
- 调用工具前要优先判断是否真的必要
- 工具结果返回后再组织最终回答

### 第一版示例

```
你当前可以使用外部工具来辅助回答问题。

请遵循以下规则：
1. 工具调用是可选的，不是必须的。
2. 如果当前对话上下文和知识库内容已经足够回答，请不要调用工具。
3. 只有在需要外部信息、外部执行能力或当前上下文不足时，才调用工具。
4. 不要为了展示能力而调用工具。
5. 获取工具结果后，再结合用户问题输出最终答案。
```

------

## 5.5 MCP 工具选择 Prompt

### 作用

帮助模型理解“当前有哪些工具”和“这些工具分别适合干什么”。

### 解决的问题

- 工具多了以后模型不会选
- 工具职责边界不清晰
- MCP 热插拔后模型难以形成稳定认知

### 第一版建议

由后端动态拼接：

- 工具名
- 工具描述
- 典型用途
- 是否高风险

### 第一版模板示例

```
当前可用工具如下：

{tool_descriptions}

在选择工具时，请遵循以下原则：
1. 优先选择与用户问题最直接相关的工具。
2. 如果多个工具都可以完成任务，优先选择作用范围更小、风险更低的工具。
3. 如果问题不需要工具，不要调用任何工具。
4. 不要调用与你无法解释用途的工具。
```

### `tool_descriptions` 示例

```
1. web_search：用于搜索网络公开信息，适合查询最新资讯、官方文档、公开网页内容。
2. file_ops：用于读取或列出指定文件内容，适合分析本地文档或项目文件。
3. browser_fetch：用于获取指定网页内容，适合打开已知链接并提取信息。
```

------

## 5.6 输出格式约束 Prompt

### 作用

控制前端看到的回答结构。

### 解决的问题

- 回答风格不稳定
- 前端需要更规整的展示
- 需要区分最终回答与中间过程

### 第一版建议

第一版不要一上来要求严格 JSON 输出。
 因为你有 Streaming + Trace，最终回答不需要再强行全 JSON 化。

更推荐的方式是要求：

- 回答结构清晰
- 尽量分点
- 如果使用了知识库或工具，可以自然说明

### 第一版示例

```
请输出适合直接展示给用户的最终回答。

要求：
1. 回答清晰、自然、可读，不要输出内部推理过程。
2. 如果使用了知识库内容，可在回答中自然说明“根据当前知识库内容”。
3. 如果使用了工具结果，可在回答中自然说明“根据工具返回结果”。
4. 不要输出技术性中间控制字段，不要输出 JSON 包裹的最终答案。
```

## 6. 四种模式下的 Prompt 组合建议

你的系统有四种模式，建议这样组合：

------

## 6.1 纯聊天模式

组合：

- `system-base`
- `chat-base`
- `answer-format`

### 目的

让模型只根据会话上下文回答，不做检索、不做工具调用。

------

## 6.2 RAG 模式

组合：

- `system-base`
- `rag-context`
- `answer-format`

### 目的

强调“优先依据知识库结果回答”。

------

## 6.3 Agent 模式

组合：

- `system-base`
- `agent-tool-policy`
- `mcp-tool-selection`
- `answer-format`

### 目的

强调“必要时调用工具，不要乱调”。

------

## 6.4 RAG + Agent 模式

组合：

- `system-base`
- `rag-context`
- `agent-tool-policy`
- `mcp-tool-selection`
- `answer-format`

### 目的

这是你的主战场模式。
 RAG 负责补上下文，Agent 负责必要时调用外部工具。

------

## 7. Prompt 变量设计建议

建议后端统一用变量注入，而不是拼接大字符串时乱塞内容。

第一版常见变量：

- `{user_question}`
- `{conversation_history}`
- `{selected_knowledge_bases}`
- `{rag_snippets}`
- `{tool_descriptions}`
- `{current_mode}`
- `{system_constraints}`

------

## 8. 第一版 Prompt 存放建议

你现在不用一开始就做“Prompt 管理平台”。

建议第一版先这样：

### 方案 A：代码中维护模板

适合当前阶段，简单直接。

例如：

- `SystemPrompts.java`
- `RagPrompts.java`
- `ToolPrompts.java`

### 方案 B：配置文件维护

适合少量 prompt，便于改文案。

### 当前建议

**先代码里维护模板常量**。
 因为你现在还在快速迭代，写数据库模板管理反而会增加复杂度。

### 当前主链路代码落地建议（2026-03-30）

如果要把 Prompt 真正固定到当前主链路代码里，建议先收口到下面几个明确落点，而不是继续让 Prompt 分散在不同运行时里各写一段：

#### 1. Prompt 先收口成代码模板目录

第一版不要做 Prompt 管理后台，建议先在后端代码里集中维护：

- `chat/prompt/PromptTemplateKey`
- `chat/prompt/PromptTemplateCatalog`
- `chat/prompt/PromptAssembly`

目标不是发明一套复杂 DSL，而是把“当前有哪些 Prompt 块、每块负责什么、最终这轮拼了哪些块”固定下来。

#### 2. 第一版固定 5 块 Prompt

当前主链路建议优先固定这 5 块，而不是一开始把所有 Prompt 变体都落完：

- `system-base`
- `rag-context`
- `agent-tool-policy`
- `mcp-tool-selection`
- `answer-format`

`chat-base` 可以继续保留在文档里，但第一版不一定要先单独做成代码块；纯聊天模式先复用 `system-base + answer-format` 也能成立。

#### 3. `ChatContextAssembler` 作为统一装配入口

当前代码里：

- `ChatContextAssembler` 还在手工拼 RAG `SystemMessage`
- `RagRuntimeResolver` 里还保留了一段局部 `PromptTemplate`
- Agent 路径目前主要是 `@UserMessage("{{message}}")`

第一版建议把 Prompt 装配职责统一收敛到 `ChatContextAssembler`：

- 负责组装系统层 Prompt
- 负责组装模式层 Prompt
- 负责承接会话层和请求层补充
- 负责产出“本轮 Prompt 由哪些块组成”的摘要

这样后面无论同步、流式、RAG 还是 Agent，都不再各自偷偷拼字符串。

#### 4. LangChain4j 官方能力优先使用

Prompt 固化这件事本身不需要绕开 LangChain4j 主线，第一版建议继续优先使用：

- `SystemMessage`
- `PromptTemplate`
- `DefaultRetrievalAugmentor`
- `AI Services`

项目自定义部分只保留在“Prompt 模板目录、Prompt 组合结果、Trace 摘要”这层，不额外发明新的模型调用抽象。

#### 5. 当前落地进展与 Trace 可见性

当前主链路已经按下面顺序落地：

1. 已把 5 块 Prompt 模板收口到 `PromptTemplateKey / PromptTemplateCatalog / PromptAssembly`。
2. 已让 `ChatContextAssembler` 产出统一的 Prompt 组合结果，并覆盖非 Agent 主链路。
3. 已把 `RagRuntimeResolver` 的局部 `PromptTemplate` 回收到统一 Prompt 目录。
4. 已给 Agent 路径补上 `system-base + agent-tool-policy + mcp-tool-selection + answer-format` 的正式注入入口。
5. 已补上 `PROMPT_ASSEMBLY_RESOLVED` trace 事件，当前 Trace / 调试视图可以看到：
   - 本轮最终使用了哪些 Prompt 块
   - Prompt 来自 `ChatContextAssembler` 还是 `AgentAiServiceFactory`
   - 各块变量的结构化摘要
   - 非 Agent 路径最终请求是否按 `system -> history -> user` 组织

这里仍然保持一个重要边界：

- Trace 默认只暴露 Prompt 组成摘要，不直接暴露整段 Prompt 原文。
- `PromptAssembly.summary()` 里的变量信息只保留长度、行数、集合大小等摘要，避免把完整 RAG 片段或系统提示全文无控制暴露到普通页面。

#### 6. 第一版验收重点

- 同一模式下回答风格和边界更稳定
- RAG 模式明确优先依赖检索结果
- Agent 模式明确工具使用边界
- 后续定位问题时，能区分是 Prompt 组合问题、RAG 问题还是工具问题

------

## 9. Prompt 调优优先级建议

不要一开始同时调所有 Prompt。
 建议优先级如下：

### 第一优先级

- 全局系统 Prompt
- RAG Prompt
- Tool 调用约束 Prompt

### 第二优先级

- 工具选择 Prompt
- 回答格式 Prompt

### 第三优先级

- 风格润色类 Prompt
- 更复杂的格式控制 Prompt

------

## 10. 当前阶段不建议做的事情

这些内容先别急着做：

- 复杂的 Prompt 管理后台
- Prompt 版本审批流
- 多层级 Prompt 继承体系
- 强制结构化 JSON 最终输出
- 过多风格化人格设定
- 过长的“行为守则大全”

因为第一版最重要的是：

- 稳定
- 可控
- 可调试

------

## 11. 第一版最小 Prompt 集合

如果你现在想立刻开工，建议先把下面 5 个固定到主链路代码里：

- 全局系统 Prompt
- RAG Prompt
- Tool 调用约束 Prompt
- MCP 工具选择 Prompt
- 最终回答格式 Prompt

这 5 个已经足够支撑：

- 纯聊天
- RAG
- Agent
- RAG + Agent

------

## 12. 一句话总结

当前 Prompt 设计策略建议是：

**用“全局角色 + 场景补充 + 输出约束”的方式组织 Prompt，先把 RAG 和 Tool 调用控制住，再慢慢优化回答效果。**
