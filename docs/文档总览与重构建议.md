- 工程骨架采用 `Spring Boot 3.5.11`

## 2. 当前文档体系的优点

- 已经明确围绕 `RAG + Agent + MCP + Streaming + Trace + Memory` 进行系统设计。
- 已经把平台级能力考虑进来了，不只是一个简单的聊天 Demo。

## 3. 当前文档体系的主要问题

虽然内容已经比较全，但当前问题也很明显。

### 3.1 文档主线更偏“平台实现”，还没有完全围绕“LangChain4J 学习”组织

你现在的核心目标不是单纯做一个 AI 平台，而是：

- 使用 `LangChain4J`
- 实现 `RAG + Agent`
- 在实现过程中沉淀知识点

但现有文档更像“我要做一个平台，所以我要把架构、数据库、接口都设计完”，而不是“我要通过这个项目系统学习 LangChain4J，所以文档要围绕 LangChain4J 的能力地图展开”。

这会带来一个问题：

- 平台设计越来越完整
- 但 LangChain4J 的学习脉络不够突出

### 3.2 RAG、Agent、MCP 热插拔之间的边界虽然有了，但学习重点还不够聚焦

你真正要掌握的不是“系统里有什么模块”，而是：

- LangChain4J 的 `ChatModel / StreamingChatModel`
- LangChain4J 的 `AI Services`
- LangChain4J 的 `Tools`
- LangChain4J 的 `ChatMemory`
- LangChain4J 的 `RAG APIs`
- LangChain4J 与 `MCP` 的集成方式

这些能力现在分散在多份文档中，没有形成“从基础到组合”的学习路径。

### 3.3 运行模式与路由策略需要有固定入口

你现在的系统不是单一路径，而是四种模式：

- 纯 LLM
- RAG
- Agent
- RAG + Agent

这是整个系统最关键的控制面之一，但目前它散落在架构、Prompt、接口等文档里，还没有成为一份独立的核心说明。

### 3.4 LangChain4J 学习主线需要单独成文

这是当前最应该补的一份文档。

因为你的目标里明确包含：

- 学会 LangChain4J
- 总结知识点

如果没有一份独立文档去记录：

- LangChain4J 提供了哪些核心能力
- 这些能力在本项目中分别落在什么位置
- 第一步先学什么，第二步再学什么
- 哪些能力当前先不用

那项目很容易越做越复杂，但学习沉淀不足。

## 4. 结合你当前目标后的主线判断

结合你补充的需求，当前项目主线应该明确收敛为：

**以 LangChain4J 为主线，构建一个支持模式切换的 RAG + Agent 学习型平台。**

这个主线下，系统不应该被表述为“一个大而全平台”，而应该被表述为下面四层：

### 4.1 第一层：基础问答

默认能力：

`用户问题 -> LLM 生成答案`

对应学习重点：

- ChatModel
- StreamingChatModel
- AI Services 基础用法

### 4.2 第二层：RAG 能力

当打开 RAG 开关时：

`用户问题 -> embedding -> Milvus -> 检索文档 -> LLM 生成答案`

对应学习重点：

- EmbeddingModel
- Document / TextSegment / Metadata
- Retriever / ContentInjector
- RAG 上下文注入

### 4.3 第三层：Agent 能力

当打开 Agent 开关时：

`用户问题 -> Agent -> Tool -> 结果 -> LLM 汇总`

对应学习重点：

- Tools
- Tool calling
- MCP 集成
- 模型自主决策是否调用工具

### 4.4 第四层：RAG + Agent 组合能力

当 RAG 和 Agent 同时打开时：

`用户问题 -> RAG 检索 -> 检索结果进入上下文 -> LLM 直接回答或调用 Tool -> 汇总结果`

对应学习重点：

- RAG 与 Tool calling 的协同
- Prompt 分层
- 模式编排
- Trace 与可观测性

这四层就是你整个项目真正的能力地图，也是后续文档应该围绕的主线。
