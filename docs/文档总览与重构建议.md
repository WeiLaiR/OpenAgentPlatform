# docs 文档总览与当前判断

更新日期：2026-03-31

## 1. 当前结论

`docs/` 目录的主干内容现在**已经基本齐了**。

如果只看“是否已经覆盖项目需要的核心主题”，当前答案基本是：**差不多了，已经不是缺文档的问题。**

现在更需要做的是两件事：

- 继续维护现有文档与代码现状的一致性
- 把入口说明、文档边界和当前阶段判断写得更清楚，减少重复和交叉覆盖

换句话说，`docs/` 现在的主要矛盾已经从“文档不够”切换成“文档已经够多，需要收口和对齐”。

## 3. 当前 docs 的主要问题

### 3.1 入口判断还不够直接

新进入仓库的人仍然需要自己判断：

- 哪些文档是“当前仍然有效的主线”
- 哪些文档更偏“设计阶段的展开说明”
- 哪些文档应该与 `status/` 配合着看

这不是文档数量问题，而是**入口和阅读顺序问题**。

### 3.2 有些文档仍保留较强的“设计阶段”写法

当前项目已经有不少能力真正落地了，但部分文档仍然偏“准备怎么做”，而不是清楚地区分：

- 当前实现
- 目标基线
- 还没做的部分

后续继续维护时，应该优先把这三层边界写清楚，而不是继续加新文档。

### 3.3 `status/` 与 `docs/` 的职责边界还可以再明确

现在最容易出现的重复是：

- `status/` 里写了很多能力细节
- `docs/` 里也写了类似内容
- 二者一旦不同步，就容易让人误把计划当现状，或误把现状当目标

后续应坚持下面这个分工：

- `status/`：写“现在已经是什么”和“现在最该做什么”
- `docs/`：写“为什么这样设计”和“目标基线是什么”

## 4. 当前对 docs 完整度的判断

如果把“完整度”拆成三层来看：

### 4.1 主题覆盖度

已经较高。

项目涉及的主线能力 `Chat / Memory / RAG / MCP / Agent / Prompt / Trace / 配置 / 数据库 / 接口 / 启动` 都已经有文档落点。

### 4.2 当前可用度

已经够用。

对于当前这版代码和协作方式来说，`docs/` 已足以支撑：

- 新人理解项目主线
- 对照状态文档判断当前阶段
- 本地启动和联调
- 后续继续沿 LangChain4j 主线推进

### 4.3 仍需持续优化的地方

还没有完全结束，主要集中在“收口”和“对齐”：

- 把文档写法从“设计草案感”继续推进到“当前实现 vs 目标基线”的明确对照
- 继续统一技术基线表述：`Spring Boot 3.5.11 + LangChain4j 官方 Spring Boot 3.5 集成口径`
- 继续减少 `status/` 与 `docs/` 之间的重复说明
- 在 Agent 风控与 observability 真正落地后，同步把对应设计文档补齐到现状

## 5. 当前推荐的阅读顺序

建议按下面顺序阅读，而不是在 `docs/` 里随机跳：

1. `status/当前实现清单.md`
2. `status/当前优先实现清单.md`
3. `docs/项目实现架构.md`
4. `docs/LangChain4J 学习路线与知识地图.md`
5. `docs/模式编排与调用链设计.md`
6. `docs/本地启动与联调说明.md`
7. 需要专题细节时，再进入数据库、接口、配置、Prompt、MCP 相关文档

## 6. 当前推荐的维护策略

### 6.1 不优先新增文档，优先更新现有文档

只有在下面两种情况，才建议新增专题文档：

- 现有文档确实装不下，继续写会让主题失焦
- 新能力已经足够独立，并且会长期演进

否则优先更新现有文档，不要让 `docs/` 再长出更多平行入口。

### 6.2 继续把 `status/` 当作状态入口

当前最适合承担“进度入口”的不是再新增一份周报式文档，而是继续维护：

- `status/当前实现清单.md`
- `status/当前优先实现清单.md`

### 6.3 代码变了，文档要同步变

尤其是下面这些改动，不能只改代码不改文档：

- 技术基线变化
- 模式编排变化
- 接口契约变化
- DDL 和表结构变化
- Prompt 主链路变化
- MCP 运行约束变化
- Trace 事件模型变化

## 7. 我对当前 docs 的判断

一句话结论：

**`docs/` 现在已经完成得差不多了，当前重点不是补更多目录，而是继续把入口、边界和现状对齐做扎实。**
