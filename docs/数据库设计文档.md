# Open Agent Platform (Java) 数据库设计文档

## 1. 设计目标

数据库需要支撑以下能力：

1. 会话与消息持久化
2. LangChain4j ChatMemory 持久化
3. 多知识库管理、文件管理、分段管理
4. MCP Server 注册、工具快照、启停控制
5. Agent 执行链路追踪（Trace）
6. 后续扩展：Prompt 模板、Agent Profile、权限控制、评测记录

本设计优先遵循以下原则：

- **MySQL 负责业务真相**
- **Milvus 负责向量检索，不承担业务主数据**
- **前端看到的是 history，模型吃的是 memory**
- **所有“动态能力”都要可配置、可追踪、可审计**
- **DDL SQL 统一在仓库内可追踪维护**

------

# 1.1 DDL 管理约定

第一版 DDL 不强依赖额外迁移框架，统一采用手工 SQL 脚本管理。

建议目录：

```
backend/sql/ddl/
```

建议约定：

- 建表、索引、初始化数据分文件维护
- 文件名按时间或阶段排序，例如 `V1__init_tables.sql`
- 文档中的表结构一旦变更，必须同步更新对应 SQL
- 只有已经实现并验证通过的结构，才写入 `status/` 文档

------

# 2. 命名规范

建议统一使用：

- 表名：`snake_case`
- 主键：`bigint unsigned`
- 时间字段：`datetime(3)`
- JSON 扩展字段：`json`
- 状态字段：`tinyint` 或 `varchar(32)`，推荐枚举值固定
- 逻辑删除：第一版**不建议全面使用**，对审计型表直接保留全量记录

字符集：

```
utf8mb4
utf8mb4_unicode_ci
```

# 4. 分库分表建议

第一版不分库分表，单库即可。

建议按业务域分模块，而不是物理拆库：

- 会话域
- 知识库域
- MCP 域
- Memory 域
- Trace 域

当出现以下情况再考虑拆分：

- `conversation_message` 过亿
- `trace_event` 写入量过高
- 多租户数据隔离要求提升

# 6. 用户与会话域

## 6.1 `user_account`

用户表。即使第一版只有单用户，也建议先建。

```
CREATE TABLE user_account (
  id BIGINT UNSIGNED PRIMARY KEY AUTO_INCREMENT,
  username VARCHAR(64) NOT NULL,
  display_name VARCHAR(128) DEFAULT NULL,
  email VARCHAR(128) DEFAULT NULL,
  status TINYINT NOT NULL DEFAULT 1 COMMENT '1=ACTIVE,0=DISABLED',
  created_at DATETIME(3) NOT NULL DEFAULT CURRENT_TIMESTAMP(3),
  updated_at DATETIME(3) NOT NULL DEFAULT CURRENT_TIMESTAMP(3) ON UPDATE CURRENT_TIMESTAMP(3),
  UNIQUE KEY uk_user_account_username (username),
  UNIQUE KEY uk_user_account_email (email)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci;
```

### 字段说明

- `status`：用户可用状态
- `username`：后续做登录、审计、资源隔离都会用到

------

## 6.2 `conversation`

会话主表。

```
CREATE TABLE conversation (
  id BIGINT UNSIGNED PRIMARY KEY AUTO_INCREMENT,
  user_id BIGINT UNSIGNED NOT NULL,
  title VARCHAR(255) DEFAULT NULL,
  mode_code VARCHAR(32) NOT NULL DEFAULT 'CHAT' COMMENT 'CHAT/RAG/AGENT/RAG_AGENT',
  enable_rag TINYINT NOT NULL DEFAULT 0,
  enable_agent TINYINT NOT NULL DEFAULT 0,
  memory_enabled TINYINT NOT NULL DEFAULT 1,
  system_prompt_id BIGINT UNSIGNED DEFAULT NULL,
  agent_profile_id BIGINT UNSIGNED DEFAULT NULL,
  status TINYINT NOT NULL DEFAULT 1 COMMENT '1=ACTIVE,0=ARCHIVED,-1=DELETED',
  last_message_at DATETIME(3) DEFAULT NULL,
  ext_json JSON DEFAULT NULL,
  created_at DATETIME(3) NOT NULL DEFAULT CURRENT_TIMESTAMP(3),
  updated_at DATETIME(3) NOT NULL DEFAULT CURRENT_TIMESTAMP(3) ON UPDATE CURRENT_TIMESTAMP(3),
  KEY idx_conversation_user_id (user_id),
  KEY idx_conversation_last_message_at (last_message_at),
  KEY idx_conversation_status_created_at (status, created_at)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci;
```

### 设计说明

- `enable_rag`、`enable_agent`：前端两个开关的持久化
- `mode_code`：冗余字段，便于查询和统计
- `memory_enabled`：允许后续做“此会话不记忆”的模式
- `ext_json`：保存 UI 偏好、模型参数快照等

### 状态流转

- `ACTIVE`：正常会话
- `ARCHIVED`：前端归档
- `DELETED`：软删除标记

------

## 6.3 `conversation_message`

完整会话历史表。
 这是 **history**，不是 memory。

```
CREATE TABLE conversation_message (
  id BIGINT UNSIGNED PRIMARY KEY AUTO_INCREMENT,
  conversation_id BIGINT UNSIGNED NOT NULL,
  user_id BIGINT UNSIGNED NOT NULL,
  role_code VARCHAR(32) NOT NULL COMMENT 'SYSTEM/USER/ASSISTANT/TOOL',
  message_type VARCHAR(32) NOT NULL COMMENT 'TEXT/TOOL_CALL/TOOL_RESULT/RAG_CONTEXT',
  content MEDIUMTEXT,
  content_json JSON DEFAULT NULL,
  request_id VARCHAR(64) DEFAULT NULL,
  parent_message_id BIGINT UNSIGNED DEFAULT NULL,
  token_count INT DEFAULT NULL,
  model_name VARCHAR(128) DEFAULT NULL,
  finish_reason VARCHAR(64) DEFAULT NULL,
  error_code VARCHAR(64) DEFAULT NULL,
  error_message VARCHAR(512) DEFAULT NULL,
  created_at DATETIME(3) NOT NULL DEFAULT CURRENT_TIMESTAMP(3),
  KEY idx_conv_msg_conversation_id_created_at (conversation_id, created_at),
  KEY idx_conv_msg_request_id (request_id),
  KEY idx_conv_msg_parent_message_id (parent_message_id),
  KEY idx_conv_msg_role_code (role_code)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci;
```

### 字段说明

- `role_code`：
  - `SYSTEM`
  - `USER`
  - `ASSISTANT`
  - `TOOL`
- `message_type`：
  - `TEXT`
  - `TOOL_CALL`
  - `TOOL_RESULT`
  - `RAG_CONTEXT`
- `content`：普通文本内容
- `content_json`：复杂结构化消息，建议存：
  - 工具调用参数
  - 检索片段列表
  - streaming 汇总
- `parent_message_id`：用于关联 tool call 和 tool result
- `request_id`：一次问答链路的请求号

### 为什么要单独建这张表

LangChain4j 官方区分了 history 和 memory；history 是 UI 看到的全部对话，memory 是给模型看的经过裁剪/处理的上下文。
这里再补一条边界：**trace 不是 history**。`conversation_message` 只保存用户可重放的消息，不保存内部执行事件。

# 7. Trace 域

## 7.1 `trace_event`

完整链路追踪表。

```
CREATE TABLE trace_event (
  id BIGINT UNSIGNED PRIMARY KEY AUTO_INCREMENT,
  conversation_id BIGINT UNSIGNED NOT NULL,
  request_id VARCHAR(64) NOT NULL,
  message_id BIGINT UNSIGNED DEFAULT NULL,
  event_type VARCHAR(64) NOT NULL,
  event_stage VARCHAR(64) DEFAULT NULL,
  event_source VARCHAR(32) NOT NULL DEFAULT 'APP_CUSTOM' COMMENT 'LANGCHAIN4J_NATIVE/APP_CUSTOM',
  event_payload_json JSON DEFAULT NULL,
  success_flag TINYINT NOT NULL DEFAULT 1,
  cost_millis INT DEFAULT NULL,
  created_at DATETIME(3) NOT NULL DEFAULT CURRENT_TIMESTAMP(3),
  KEY idx_trace_conversation_id_created_at (conversation_id, created_at),
  KEY idx_trace_request_id_created_at (request_id, created_at),
  KEY idx_trace_event_type_created_at (event_type, created_at)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci;
```

### 建议事件类型

- `USER_MESSAGE_RECEIVED`
- `MEMORY_LOADED`
- `RAG_STARTED`
- `RAG_FINISHED`
- `RAG_SEGMENTS_SELECTED`
- `TOOLS_ASSEMBLED`
- `TOOL_CALL_STARTED`
- `TOOL_CALL_FINISHED`
- `MODEL_STREAMING`
- `ANSWER_COMPLETED`
- `ERROR_OCCURRED`

### 事件来源建议

- `LANGCHAIN4J_NATIVE`：直接映射 LangChain4j observability / listener 事件
- `APP_CUSTOM`：平台特有事件，例如会话锁竞争、人工确认结果、MCP 运行时工具装配结果

### `event_payload_json` 示例

```
{
  "runtimeToolName": "search_server__web_search",
  "originToolName": "web_search",
  "args": {"query": "langchain4j mcp"},
  "resultSummary": "返回3条结果"
}
```

### 为什么 trace 独立成表

因为它天然是**事件流**，和用户消息不是一回事。
 把 trace 混进 `conversation_message` 会导致 UI 展示、审计查询、统计分析都变复杂。

# 8. 知识库域

## 8.1 `knowledge_base`

知识库主表。

```
CREATE TABLE knowledge_base (
  id BIGINT UNSIGNED PRIMARY KEY AUTO_INCREMENT,
  owner_user_id BIGINT UNSIGNED NOT NULL,
  name VARCHAR(128) NOT NULL,
  description VARCHAR(512) DEFAULT NULL,
  status TINYINT NOT NULL DEFAULT 1 COMMENT '1=ACTIVE,0=DISABLED',
  embedding_model_name VARCHAR(128) NOT NULL,
  embedding_dimension INT NOT NULL,
  milvus_database_name VARCHAR(128) DEFAULT NULL,
  milvus_collection_name VARCHAR(128) NOT NULL,
  milvus_partition_name VARCHAR(128) NOT NULL,
  parser_strategy VARCHAR(64) DEFAULT 'TIKA',
  chunk_strategy VARCHAR(64) DEFAULT 'DEFAULT',
  chunk_size INT NOT NULL DEFAULT 1000,
  chunk_overlap INT NOT NULL DEFAULT 150,
  ext_json JSON DEFAULT NULL,
  created_at DATETIME(3) NOT NULL DEFAULT CURRENT_TIMESTAMP(3),
  updated_at DATETIME(3) NOT NULL DEFAULT CURRENT_TIMESTAMP(3) ON UPDATE CURRENT_TIMESTAMP(3),
  UNIQUE KEY uk_kb_name_owner (owner_user_id, name),
  KEY idx_kb_status_created_at (status, created_at),
  KEY idx_kb_collection_partition (milvus_collection_name, milvus_partition_name)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci;
```

### 设计说明

- `milvus_collection_name` + `milvus_partition_name`：直接映射 Milvus
- `milvus_database_name`：第一版建议固定为 `openagent`
- `milvus_collection_name` + `milvus_partition_name`：第一版建议固定为 `knowledge_segment` + `kb_{knowledgeBaseId}`
- `embedding_model_name` + `embedding_dimension`：避免后期混乱
- `chunk_size`、`chunk_overlap`：知识库级别可配置
- 第一版可先采用字符维度分块；第二阶段可升级为 recursive token splitter

### 为什么要保存 partition

你当前要支持“多个知识库多选联合检索”，第一版最合理的就是一个 collection 下多个 partition。Milvus 官方把 partition 作为多租户策略之一，适合在隔离和灵活查询之间取平衡。
但这里应明确：这是 **v1 方案**，未来知识库规模增长后可升级为 metadata filter 或更细的 collection 规划。

## 8.3 `knowledge_segment`

文档分段元数据表。

```
CREATE TABLE knowledge_segment (
  id BIGINT UNSIGNED PRIMARY KEY AUTO_INCREMENT,
  knowledge_base_id BIGINT UNSIGNED NOT NULL,
  file_id BIGINT UNSIGNED NOT NULL,
  segment_no INT NOT NULL,
  text_preview VARCHAR(1024) DEFAULT NULL,
  full_text MEDIUMTEXT,
  token_count INT DEFAULT NULL,
  page_no INT DEFAULT NULL,
  source_title VARCHAR(255) DEFAULT NULL,
  source_path VARCHAR(512) DEFAULT NULL,
  metadata_json JSON DEFAULT NULL,
  milvus_primary_key VARCHAR(128) NOT NULL,
  created_at DATETIME(3) NOT NULL DEFAULT CURRENT_TIMESTAMP(3),
  UNIQUE KEY uk_file_segment_no (file_id, segment_no),
  UNIQUE KEY uk_segment_milvus_pk (milvus_primary_key),
  KEY idx_ks_kb_file_id (knowledge_base_id, file_id),
  KEY idx_ks_page_no (page_no)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci;
```

### 为什么 MySQL 里也要存 segment

因为 Milvus 只擅长向量检索，不适合作为业务元数据真相。
 MySQL 中保留 `segment` 后，你可以做：

- 前端展示命中文档片段
- 删除/重建索引
- 片段级审计
- 根据文件重建向量

### `metadata_json` 建议内容

```
{
  "docType": "pdf",
  "sectionTitle": "安装指南",
  "chunkStart": 1024,
  "chunkEnd": 1980
}
```

### `milvus_primary_key` 建议

- 该字段建议由应用层提前生成，而不是依赖 MySQL 自增主键回填
- 推荐格式：`ks_{knowledgeBaseId}_{fileId}_{segmentNo}`
- 该值应与 Milvus 主键字段值完全一致，用于 MySQL / Milvus 双向关联

# 9. MCP 域

## 9.1 `mcp_server`

MCP 服务注册表。

```
CREATE TABLE mcp_server (

  enabled TINYINT NOT NULL DEFAULT 1,
  health_status VARCHAR(32) NOT NULL DEFAULT 'UNKNOWN' COMMENT 'UNKNOWN/HEALTHY/UNHEALTHY',
  risk_level VARCHAR(32) NOT NULL DEFAULT 'MEDIUM' COMMENT 'LOW/MEDIUM/HIGH',
  ext_json JSON DEFAULT NULL,
  last_connected_at DATETIME(3) DEFAULT NULL,
  last_sync_at DATETIME(3) DEFAULT NULL,

  KEY idx_mcp_transport_type (transport_type)

```

### 设计说明

- `transport_type`：适配 `STDIO` / `STREAMABLE_HTTP`
- `risk_level`：后续可用于高风险工具调用确认
- `health_status`：前端状态灯、后台巡检都会用到

### 为什么要做风险等级

MCP 规范强调工具应保留人工控制与拒绝执行能力，因此平台级最好从一开始就把安全分级建进去。

## 9.2 `mcp_tool_snapshot`

MCP 工具快照表。

```
CREATE TABLE mcp_tool_snapshot (
  id BIGINT UNSIGNED PRIMARY KEY AUTO_INCREMENT,
  mcp_server_id BIGINT UNSIGNED NOT NULL,
  runtime_tool_name VARCHAR(128) NOT NULL COMMENT '运行时全局唯一名，如 serverAlias__toolName',
  origin_tool_name VARCHAR(128) NOT NULL COMMENT '远端 MCP 原始工具名',
  tool_title VARCHAR(255) DEFAULT NULL,
  description TEXT,
  input_schema_json JSON DEFAULT NULL,
  output_schema_json JSON DEFAULT NULL,
  enabled TINYINT NOT NULL DEFAULT 1,
  risk_level VARCHAR(32) NOT NULL DEFAULT 'MEDIUM',
  version_no VARCHAR(64) DEFAULT NULL,
  sync_hash VARCHAR(128) DEFAULT NULL,
  synced_at DATETIME(3) NOT NULL DEFAULT CURRENT_TIMESTAMP(3),
  created_at DATETIME(3) NOT NULL DEFAULT CURRENT_TIMESTAMP(3),
  updated_at DATETIME(3) NOT NULL DEFAULT CURRENT_TIMESTAMP(3) ON UPDATE CURRENT_TIMESTAMP(3),
  UNIQUE KEY uk_mcp_runtime_tool (runtime_tool_name),
  UNIQUE KEY uk_mcp_server_origin_tool (mcp_server_id, origin_tool_name),
  KEY idx_mcp_tool_enabled (enabled),
  KEY idx_mcp_tool_synced_at (synced_at)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci;
```

### 为什么需要 snapshot

因为 MCP 的 tool 是动态发现的，但你前端需要稳定展示：

- 有哪些工具
- 描述是什么
- 是否启用
- 上次同步时间
- schema 长什么样

MCP 规范里工具带有名称和 schema 元数据，正适合本地快照化。
但一旦你把多个 server 的工具合并到同一个模型上下文中，平台就必须负责运行时全局唯一命名，因此这里要同时保存：

- `origin_tool_name`：原始协议名，便于同步和排障
- `runtime_tool_name`：运行时全局唯一名，便于装配给模型

# 10. Memory 域

## 10.1 `chat_memory_session`

Memory 会话表。

```
CREATE TABLE chat_memory_session (
  id BIGINT UNSIGNED PRIMARY KEY AUTO_INCREMENT,
  conversation_id BIGINT UNSIGNED NOT NULL,
  memory_id VARCHAR(128) NOT NULL,
  memory_type VARCHAR(32) NOT NULL DEFAULT 'TOKEN_WINDOW' COMMENT 'MESSAGE_WINDOW/TOKEN_WINDOW',
  max_messages INT DEFAULT NULL,
  max_tokens INT DEFAULT NULL,
  tokenizer_name VARCHAR(128) DEFAULT NULL,
  status TINYINT NOT NULL DEFAULT 1,
  created_at DATETIME(3) NOT NULL DEFAULT CURRENT_TIMESTAMP(3),
  updated_at DATETIME(3) NOT NULL DEFAULT CURRENT_TIMESTAMP(3) ON UPDATE CURRENT_TIMESTAMP(3),
  UNIQUE KEY uk_memory_session_memory_id (memory_id),
  UNIQUE KEY uk_memory_session_conversation_id (conversation_id)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci;
```

### 设计说明

- `memory_id`：直接映射 LangChain4j ChatMemory 的 id
- `memory_type`：后续你可以从 `TokenWindow` 换成别的策略
- 推荐第一版用 `TOKEN_WINDOW`

LangChain4j 官方说明里，`MessageWindowChatMemory` 更适合快速原型，而 `TokenWindowChatMemory` 更适合按 token 控制上下文窗口。

------

## 10.2 `chat_memory_message`

Memory 消息表。
 注意：这是“给模型使用的 memory 消息”，不是完整 history。

```
CREATE TABLE chat_memory_message (
  id BIGINT UNSIGNED PRIMARY KEY AUTO_INCREMENT,
  memory_session_id BIGINT UNSIGNED NOT NULL,
  message_order INT NOT NULL,
  role_code VARCHAR(32) NOT NULL,
  message_json JSON NOT NULL,
  token_count INT DEFAULT NULL,
  created_at DATETIME(3) NOT NULL DEFAULT CURRENT_TIMESTAMP(3),
  UNIQUE KEY uk_memory_order (memory_session_id, message_order),
  KEY idx_memory_session_id (memory_session_id)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci;
```

### 为什么单独建表

LangChain4j 的 `ChatMemoryStore` 是“更新整个 memory 视图”的接口；被淘汰的消息也会从 store 中消失，所以数据库必须能明确表达“当前 memory 窗口”。

### 并发约束

同一个 `memory_id` / `conversation_id` 不应并发写入。实现上建议在 conversation 维度串行执行请求，避免多个生成任务同时覆盖 ChatMemory 视图。

### 建议实现方式

- 每次 `updateMessages(memoryId, messages)`：
  - 删除该 `memory_session_id` 下旧消息
  - 批量插入新消息
- 这种方式简单、稳定、与框架语义一致

# 12. 索引设计建议

## 12.1 高频查询路径

### 聊天页

- 按 `conversation_id + created_at` 查询消息
- 按 `conversation_id + created_at` 查询 trace

### 知识库管理页

- 按 `knowledge_base_id` 查文件
- 按 `file_id` 查 segments

### MCP 管理页

- 按 `mcp_server_id` 查 tools
- 按 `enabled + health_status` 查 server

## 12.2 索引原则

- 先覆盖主查询路径
- JSON 字段默认不建索引
- 不要一开始为了“以后可能会用”加太多组合索引

# 13. 状态机设计

## 13.1 文件解析与索引状态机

```
UPLOADED
  → PARSING
    → PARSED
      → INDEXING
        → INDEXED

任意阶段失败
  → FAILED
```

建议 `parse_status` 和 `index_status` 分开，避免状态含义混乱。

## 13.2 MCP Server 状态机

```
CREATED
  → ENABLED
  → CONNECTED
  → TOOLS_SYNCED

异常时：
  → UNHEALTHY
禁用时：
  → DISABLED
```

### 落地方式

第一版不必做单独状态字段枚举太复杂，可用：

- `enabled`
- `health_status`
- `last_connected_at`
- `last_sync_at`

# 14. 数据一致性建议

## 14.1 MySQL 与 Milvus 的一致性

Milvus 不应作为唯一真相源。

### 入库顺序建议

1. 文件元数据入 MySQL
2. 文档解析
3. segments 入 MySQL
4. 向量写入 Milvus
5. 更新文件 `index_status=INDEXED`

### 删除知识库文件时

1. 先标记业务状态
2. 删除 Milvus 向量
3. 再更新 MySQL 状态

这样即使中途失败，也能补偿。

## 14.2 MCP 工具同步一致性

`mcp_tool_snapshot` 只做“本地缓存快照”，不是协议真相。
 同步时以远端 MCP 返回工具清单为准，刷新本地表。
`description` 使用 `TEXT`，因为部分官方 MCP Server 会返回较长的工具说明，固定短 `VARCHAR` 会直接导致同步失败。

# 15. 审计与安全建议

## 15.1 建议审计的对象

- 工具调用请求参数
- 工具调用结果摘要
- 高风险工具启停
- 高风险工具确认结果
- 文件上传与索引
- 知识库删除
- MCP 配置修改

## 15.2 敏感信息处理

这些字段不要明文裸奔：

- `auth_config_json`
- `headers_json`
- `env_json`

建议：

- 数据库存加密
- 或只存引用 ID，密钥放配置中心

### 远程 MCP 额外建议

- `STREAMABLE_HTTP` 远程地址建议做 allowlist 或管理员限制
- Origin 校验与认证要求不要只停留在代码里，文档中也要明确

# 17. 最终推荐

如果只给一句落地建议，就是：

**把 MySQL 当成“平台主数据库”，把 Milvus 当成“检索执行引擎”，把 LangChain4j ChatMemory 当成“模型上下文视图”，三者职责必须严格分开。**

这会让你后面做：

- 会话重放
- RAG 片段展示
- MCP 热插拔
- Trace 调试
- Memory 优化

都非常顺。
