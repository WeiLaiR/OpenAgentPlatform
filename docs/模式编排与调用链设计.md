# 模式编排与调用链设计

## 1. 文档目标

这份文档专门回答一个核心问题：**当前系统在 `enableRag` 和 `enableAgent` 两个开关组合下，后端到底怎么决定执行链路。**

它的作用是固定运行模式、Prompt 分层、Trace 事件和实现边界，避免这部分逻辑散落在架构、接口、Prompt 文档中。

## 2. 编排总原则

### 2.1 前端只暴露两个主开关

- `enableRag`
- `enableAgent`

其余参数如 `knowledgeBaseIds`、`mcpServerIds`、`memoryEnabled`、模型参数等，都属于模式附属配置，不单独定义为“第五种模式”。

### 2.2 配置优先级固定

建议从现在起固定为：

`本次请求覆盖项 > 会话配置 > 系统默认值`

这和 LangChain4j `AI Services` 支持“按调用传入 `ChatRequestParameters` 并合并”的思路一致，适合做请求级覆盖。

### 2.3 同一会话串行执行

如果使用 `ChatMemoryProvider + @MemoryId`，官方明确不建议对同一 `@MemoryId` 并发调用 AI Service，否则可能破坏 `ChatMemory`。因此本项目应以 `conversationId` 作为串行锁粒度。

## 3. 统一运行时组件

无论哪种模式，建议都走同一条主编排骨架：

```text
ChatController
  -> ChatApplicationService
  -> ConversationLock
  -> ModeResolver
  -> ChatContextAssembler
  -> AI Service / StreamingChatModel
  -> TracePublisher
  -> SSE Publisher
```

各组件职责建议如下：

- `ConversationLock`：保证同会话串行
- `ModeResolver`：把请求参数转换成最终执行模式
- `ChatContextAssembler`：装配 memory、RAG、tools、prompt
- `TracePublisher`：统一发过程事件
- `SSE Publisher`：分别推送 answer stream 和 trace stream

## 4. 四种模式定义

## 4.1 模式 A：纯 LLM

条件：

- `enableRag=false`
- `enableAgent=false`

执行链路：

```text
用户消息
  -> 读取会话配置
  -> 加载 ChatMemory
  -> 组装基础 Prompt
  -> 调用 AI Service / StreamingChatModel
  -> 输出最终答案
```

这一模式的目的不是弱化系统，而是提供最稳定的基础链路，也是所有问题排查时的基准模式。

推荐 Trace 事件：

- `USER_MESSAGE_RECEIVED`
- `MEMORY_LOADED`
- `MODEL_REQUEST_STARTED`
- `FINAL_RESPONSE_STREAMING`
- `FINAL_RESPONSE_COMPLETED`

## 4.2 模式 B：RAG

条件：

- `enableRag=true`
- `enableAgent=false`

执行链路：

```text
用户消息
  -> 加载 ChatMemory
  -> 依据 knowledgeBaseIds 选择检索范围
  -> ContentRetriever 检索
  -> RetrievalAugmentor 注入上下文
  -> 调用 AI Service / StreamingChatModel
  -> 输出最终答案
```

实现建议：

- 第一版优先使用 `EmbeddingStoreContentRetriever`
- 用 `DefaultRetrievalAugmentor` 作为默认注入入口
- 通过 `Result<T>.sources()` 或 `TokenStream.onRetrieved()` 把命中片段送入 Trace
- 建议 `storeRetrievedContentInChatMemory(false)`，避免把大段检索上下文直接塞回会话记忆

推荐 Trace 事件：

- `USER_MESSAGE_RECEIVED`
- `MEMORY_LOADED`
- `RAG_RETRIEVAL_STARTED`
- `RAG_SEGMENTS_SELECTED`
- `RAG_RETRIEVAL_FINISHED`
- `FINAL_RESPONSE_STREAMING`
- `FINAL_RESPONSE_COMPLETED`

## 4.3 模式 C：Agent

条件：

- `enableRag=false`
- `enableAgent=true`

执行链路：

```text
用户消息
  -> 加载 ChatMemory
  -> 按会话与权限装配工具
  -> AI Service 暴露 tools / toolProvider
  -> 模型判断是否调用工具
  -> 执行工具
  -> 工具结果回送模型
  -> 汇总答案
```

实现建议：

- 平台内工具可先使用 `@Tool`
- 动态工具集使用 `ToolProvider`
- MCP 工具统一由 `McpToolProvider` 注入
- 使用 `beforeToolExecution` 与 `onToolExecuted` 输出工具调用过程

注意：

- 模型是否调用工具，交给模型决定
- 工具是否可见，交给平台决定
- 工具是否允许执行，交给权限和风控决定

推荐 Trace 事件：

- `USER_MESSAGE_RECEIVED`
- `MEMORY_LOADED`
- `TOOLS_ASSEMBLED`
- `MODEL_DECISION_STARTED`
- `TOOL_CALL_REQUESTED`
- `TOOL_CALL_FINISHED`
- `FINAL_RESPONSE_STREAMING`
- `FINAL_RESPONSE_COMPLETED`

## 4.4 模式 D：RAG + Agent

条件：

- `enableRag=true`
- `enableAgent=true`

执行链路：

```text
用户消息
  -> 加载 ChatMemory
  -> 执行 RAG 检索
  -> 将检索结果注入上下文
  -> 动态装配 tools / MCP tools
  -> 模型判断：
       1. 直接回答
       2. 调用一个或多个工具
  -> 如有工具调用，执行并回送结果
  -> 汇总答案
```

这是项目真正的目标模式。RAG 负责补充知识上下文，Agent 负责补充动作能力，二者不是替代关系。

推荐 Trace 事件：

- `USER_MESSAGE_RECEIVED`
- `MEMORY_LOADED`
- `RAG_RETRIEVAL_STARTED`
- `RAG_SEGMENTS_SELECTED`
- `RAG_RETRIEVAL_FINISHED`
- `TOOLS_ASSEMBLED`
- `MODEL_DECISION_STARTED`
- `TOOL_CALL_REQUESTED`
- `TOOL_CALL_FINISHED`
- `FINAL_RESPONSE_STREAMING`
- `FINAL_RESPONSE_COMPLETED`

## 5. Prompt 分层建议

无论哪种模式，都建议把 Prompt 分为四层，而不是在代码里一次性拼大字符串。

### 5.1 系统层 Prompt

固定角色、输出约束、安全边界。

### 5.2 模式层 Prompt

按模式补充行为规则：

- 纯 LLM：直接回答
- RAG：优先依据检索上下文回答
- Agent：必要时使用工具
- RAG + Agent：先利用上下文，再决定是否调用工具

### 5.3 会话层 Prompt

用户偏好、知识库选择、启用的 MCP server、会话级设置。

### 5.4 本次请求层 Prompt

临时覆盖项、一次性说明、本次任务目标。

## 6. 推荐调用顺序

当前项目更适合采用下面的固定顺序：

```text
先加载 memory
再做 RAG 检索
再装配 tools
最后调用模型
```

原因：

- memory 决定会话上下文基础面
- RAG 决定“知道什么”
- tools 决定“还能做什么”
- 最后统一交给模型做回答或工具调用决策

这比“先让模型判断要不要检索、再判断要不要工具”更稳，第一版也更容易观测和调试。

## 7. Streaming 与 SSE 设计

建议继续坚持“两步式 + 双流”：

1. `POST /api/v1/chat/send` 创建请求，返回 `requestId`
2. `GET /api/v1/chat/stream/{requestId}` 订阅答案流
3. `GET /api/v1/traces/stream/{requestId}` 订阅 Trace 流

与 LangChain4j 的对应关系：

- token 输出：`TokenStream.onPartialResponse`
- RAG 命中：`TokenStream.onRetrieved`
- tool 执行前：`beforeToolExecution`
- tool 执行后：`onToolExecuted`
- 完成：`onCompleteResponse`
- 错误：`onError`

如果你后续直接使用低层 `StreamingChatModel`，也可以用 `StreamingChatResponseHandler` 的 `onPartialResponse`、`onPartialToolCall`、`onCompleteToolCall` 等回调实现同样的链路。

## 8. Trace 采集建议

Trace 不建议只靠自定义埋点，应分两层：

### 8.1 LangChain4j 原生事件层

使用 `ChatModelListener` 记录：

- 最终发送给模型的 messages
- 合并后的模型参数
- 模型响应
- 错误信息

### 8.2 平台自定义事件层

补足模型外流程：

- 会话加锁/释放
- RAG 检索耗时
- MCP server 选择结果
- Tool Registry 装配结果
- 权限拒绝或人工确认

这样才能把“模型行为”和“平台行为”拆开看。

## 9. MCP 在调用链中的位置

MCP 不是新的模式，而是 Agent 模式下的一类工具来源。

推荐定位：

```text
MCP Server
  -> McpClient
  -> McpToolProvider
  -> ToolProviderResult
  -> AI Service 当前请求可见工具集
```

因此前端开关仍然保持简单：

- `enableAgent` 控制是否允许工具链路
- `mcpServerIds` 控制本次会话可见的 MCP 工具来源

## 10. 失败与降级策略

### 10.1 RAG 失败

- 检索超时或无结果时，不中断回答
- 降级为纯 LLM 或 Agent 模式
- Trace 中保留失败原因

### 10.2 工具失败

- Tool 执行异常写入 Trace
- 工具结果返回模型时标注失败信息
- 允许模型重试或改走直接回答

### 10.3 MCP 不可用

- 不让整个聊天请求失败
- 仅剔除当前不可用 server 的工具
- 在 `TOOLS_ASSEMBLED` 事件里记录可见工具变化

## 11. 第一版实现建议

第一版最稳的落地方式不是复杂 Agent 图，而是：

```text
AI Services
 + ChatMemoryProvider
 + DefaultRetrievalAugmentor
 + ToolProvider / McpToolProvider
 + TokenStream / ChatModelListener
```

这条路径和 LangChain4j 官方主线一致，也最适合当前项目“边学边做”的目标。

## 12. 最终结论

这套系统本质上不是四套后端，而是一套统一编排骨架下的四种配置模式：

- 模式 A：只回答
- 模式 B：带知识回答
- 模式 C：带工具回答
- 模式 D：带知识且可行动地回答

后续如果要继续扩展，也应优先在这套骨架上增加能力，而不是再发明新的模式名称。
