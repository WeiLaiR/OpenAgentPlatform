# Open Agent Platform (Java) 测试与验收清单

## 1. 文档目标

本文档用于定义项目开发过程中的基础测试项与阶段验收标准。

目标：

- 为每个核心模块提供最小可验证标准
- 避免“代码写完了，但其实没真正跑通”
- 帮助你在阶段开发结束后快速自检
- 为后续联调和问题定位提供依据

原则：

- 先验证主链路
- 先验证可用性，再验证完善性
- 优先覆盖核心场景
- 不追求一开始就有完整自动化测试体系

------

## 2. 测试范围

当前阶段建议覆盖以下 7 类能力：

1. 项目基础运行
2. 基础聊天能力
3. 会话与消息持久化
4. RAG 能力
5. MCP 管理与 Tool 调用
6. RAG + Agent 联合能力
7. Trace 与 Memory

------

## 3. 通用测试原则

### 3.1 先测主流程

优先验证：

- 能不能跑通
- 输出是否合理
- 前后端链路是否完整

### 3.2 先手工测试，再补自动化

当前阶段最重要的是：

- 手工把功能走通
- 明确失败点
- 记录验收标准

### 3.3 每完成一个阶段都要做一次回归

例如做完 MCP 后，至少回归：

- 纯聊天
- RAG
- 会话历史
- 前端流式输出

避免新功能把旧功能弄坏。

------

## 4. 基础运行测试

------

## 4.1 后端启动测试

### 测试项

- Spring Boot 应用能正常启动
- 无关键 Bean 注入失败
- 数据源能正常初始化
- 核心配置能正常加载

### 验收标准

- 后端启动成功
- 启动日志无明显报错
- 健康检查接口可访问

------

## 4.2 前端启动测试

### 测试项

- Vue3 项目能正常启动
- 页面可正常打开
- 基础路由可访问

### 验收标准

- 前端页面可访问
- 无明显白屏或初始化报错

------

## 4.3 前后端联通测试

### 测试项

- 前端能访问后端基础接口
- 跨域配置正常
- 请求失败时有合理提示

### 验收标准

- 基础请求成功
- 网络错误可见、可定位

------

## 5. 基础聊天测试

------

## 5.1 单轮聊天测试

### 测试项

- 用户发送消息成功
- 后端正常调用 LLM
- 前端正常展示回答

### 验收标准

- 用户至少可完成一轮问答
- 回答内容能正常显示
- 无明显阻塞或报错

------

## 5.2 Streaming 测试

### 测试项

- SSE 正常建立连接
- 回答以流式方式逐步输出
- 回答结束时能正确收尾
- `POST /api/v1/chat/send` 能返回 `requestId`
- `GET /api/v1/chat/stream/{requestId}` 可按 `requestId` 订阅
- `GET /api/v1/traces/stream/{requestId}` 可单独建立 Trace SSE

### 验收标准

- 前端能看到逐步输出
- 不会一次性全部返回才显示
- 结束状态明确
- 聊天流与 Trace 流可以解耦使用

------

## 5.3 异常场景测试

### 测试项

- 模型服务不可用
- SSE 中断
- 后端超时
- 请求参数为空
- 同一会话并发发送第二个请求

### 验收标准

- 前端能看到错误提示
- 后端不会崩溃
- 错误可通过日志或 requestId 定位
- 并发请求要么排队，要么收到明确拒绝，不允许静默覆盖 memory

------

## 6. 会话与消息持久化测试

------

## 6.1 会话创建测试

### 测试项

- 可创建新会话
- 新会话能出现在会话列表
- 新会话默认配置正确

### 验收标准

- 新建会话成功
- 列表中可见
- 配置字段保存正确

------

## 6.2 历史消息持久化测试

### 测试项

- 用户消息入库
- assistant 消息入库
- 会话刷新后历史仍可显示

### 验收标准

- 一轮对话至少保存两条消息
- 页面刷新后历史不丢失

------

## 6.3 会话切换测试

### 测试项

- 切换不同会话
- 加载对应历史消息
- 不会串会话内容

### 验收标准

- A 会话和 B 会话消息隔离正确
- 切换后显示内容正确

------

## 7. RAG 测试

------

## 7.1 文件上传测试

### 测试项

- 可上传 pdf/docx/md/txt
- 文件信息写入数据库
- 非法格式被拒绝

### 验收标准

- 支持格式上传成功
- 不支持格式有明确提示

------

## 7.2 文档解析测试

### 测试项

- 文档解析成功
- 能生成文本内容
- 能按预期分块

### 验收标准

- 文件解析状态正确
- 至少能看到 segment 结果
- 文本未出现大面积乱码或空白

------

## 7.3 向量入库测试

### 测试项

- embedding 正常生成
- 向量写入 Milvus
- segment 与 Milvus 主键能对应

### 验收标准

- 索引完成后状态正确
- Milvus 中存在对应数据
- MySQL 元数据一致

------

## 7.4 检索测试

### 测试项

- 输入问题可检索到相关片段
- 支持多个知识库多选
- 不相关问题时结果较少或为空

### 验收标准

- 检索结果与知识库内容基本相关
- 多知识库多选时结果范围正确
- 不会总是返回明显无关内容

------

## 7.5 RAG 问答测试

### 测试项

- 开启 RAG 后回答能参考知识库内容
- 未开启 RAG 时不应强依赖知识库
- 检索不到时行为合理

### 验收标准

- 开启 RAG 后回答质量明显区别于纯聊天
- 检索不足时不会伪造“知识库结论”

------

## 8. MCP 与 Tool 测试

------

## 8.1 MCP Server 创建测试

### 测试项

- 可新增 MCP Server
- 配置信息正确保存
- Server 列表可正常展示

### 验收标准

- 新建成功
- 配置回显正确
- 状态字段合理

------

## 8.2 MCP 连接测试

### 测试项

- 可对 server 发起连接测试
- 连接成功/失败状态可识别

### 验收标准

- 连接成功时状态更新
- 连接失败时能返回明确原因

------

## 8.3 Tool 同步测试

### 测试项

- 可同步工具列表
- 工具快照入库
- 工具描述/schema 可展示
- 不同 MCP server 下同名工具可同时同步成功
- 同步后能看到运行时全局唯一工具名与原始工具名

### 验收标准

- 工具数量正确
- 列表可见
- 可启用/禁用工具
- 不会因为跨 server 重名导致工具覆盖或冲突

------

## 8.4 Tool 调用测试

### 测试项

- 开启 Agent 后模型可调用工具
- 工具返回结果可被模型使用
- 工具失败时有合理处理
- 高风险工具调用前需要明确确认
- 拒绝高风险工具后模型与后端状态一致

### 验收标准

- 至少一个工具调用场景成功
- 工具结果能够反映到最终回答中
- 失败不会导致整轮对话不可用
- 高风险工具不会在无确认情况下直接执行

------

## 9. RAG + Agent 联合测试

------

## 9.1 四种模式测试

需要分别验证：

### 模式一：纯聊天

- enableRag = false
- enableAgent = false

### 模式二：RAG

- enableRag = true
- enableAgent = false

### 模式三：Agent

- enableRag = false
- enableAgent = true

### 模式四：RAG + Agent

- enableRag = true
- enableAgent = true

### 验收标准

- 四种模式都能跑通
- 行为与开关配置一致
- 不会出现开关失效或逻辑混乱

### 配置优先级补充测试

- 会话 settings 为默认关闭 RAG，请求级 `enableRag=true` 时，本次请求应启用 RAG
- 会话 settings 为开启 Agent，请求级 `enableAgent=false` 时，本次请求应禁用 Agent
- 请求结束后再次查询会话 settings，持久配置不应被单次请求覆盖

------

## 9.2 联合模式重点测试

### 测试项

- 有知识库内容时优先结合检索结果
- 需要外部能力时可进一步调用工具
- 不需要工具时不会强制调用
- 不需要检索时回答仍自然

### 验收标准

- 联合模式下系统行为合理
- RAG 与 Tool 不互相冲突
- 最终回答能结合多方上下文

------

## 10. Trace 测试

------

## 10.1 Trace 记录测试

### 测试项

- 每轮请求有 requestId
- 能记录主要事件
- 事件顺序基本正确
- LangChain4j 原生事件与平台自定义事件都能落到 trace
- trace 不会混入会话历史消息列表

### 验收标准

- 可查到一轮完整 trace
- 至少包含：
  - 用户请求
  - RAG（如果启用）
  - Tool 调用（如果触发）
  - 最终回答

------

## 10.2 前端展示测试

### 测试项

- 前端能展示 trace 列表/时间线
- 能看到 RAG 命中摘要
- 能看到工具调用摘要

### 验收标准

- 页面能直观看到本轮内部过程
- 关键事件不会丢失

------

## 11. Memory 测试

------

## 11.1 多轮上下文测试

### 测试项

- 连续多轮对话时能保持基本上下文
- 模型能引用上一轮信息
- 切换会话时 memory 不串

### 验收标准

- 同一会话上下文连续
- 不同会话隔离正确

------

## 11.2 Memory 清理测试

### 测试项

- 清空 memory 后模型不再引用旧上下文
- 历史消息仍保留（如果设计如此）

### 验收标准

- memory 与 history 行为符合设计预期

------

## 12. 数据一致性测试

------

## 12.1 MySQL 与 Milvus 一致性测试

### 测试项

- 文件索引成功后，MySQL 和 Milvus 状态一致
- 删除文件后，两边数据都被正确处理
- 重建索引后无脏数据残留

### 验收标准

- 不会出现“数据库显示已完成，但向量不存在”
- 不会出现 segment 已删但向量还在

------

## 12.2 MCP 快照一致性测试

### 测试项

- 工具同步后快照表正确更新
- 禁用工具后实际调用不可用
- 重新同步后列表正确刷新

### 验收标准

- 页面展示状态与后端实际状态一致

------

## 13. 性能与稳定性基础测试

当前阶段不需要正式压测，但建议做最小稳定性检查。

### 建议测试项

- 连续发送多轮消息是否稳定
- 大一点的文档上传是否正常
- RAG 检索响应是否明显过慢
- MCP 工具调用是否容易卡死
- SSE 长连接是否容易异常断开

### 验收标准

- 基本可连续使用
- 不出现频繁崩溃
- 出现异常时可恢复、可定位

------

## 14. 当前阶段最重要的验收清单

如果你想先抓最关键的，我建议只盯住这 10 条：

1. 前后端都能启动
2. 前端能发送消息到后端
3. SSE 流式输出正常
4. 会话和消息能入库
5. 文件可上传并解析
6. 向量可写入 Milvus
7. 开启 RAG 后回答能利用知识库
8. MCP server 可配置并同步工具
9. Agent 能成功触发至少一个工具调用
10. Trace 能记录一轮请求的关键过程

这 10 条都通过，项目主干就已经成立了。

------

## 15. 当前不必过度追求的测试内容

这些现在先不用做得太深：

- 完整自动化 UI 测试
- 高并发压测
- 大规模文档性能压测
- 多租户隔离测试
- 权限边界测试
- 部署容灾测试
- Prompt A/B 测试
- 复杂评测体系

因为你当前阶段核心是：
 **先把平台做出来并稳定跑通。**

------

## 16. 一句话总结

当前测试与验收策略建议是：

**每完成一个阶段，就围绕“主链路是否跑通、数据是否正确、前端是否可见、异常是否可定位”做一次小验收。**
