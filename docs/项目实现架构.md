# Open Agent Platform

## 1. 项目目标

构建一个基于 **Spring Boot + LangChain4j + Milvus + MySQL + Vue3** 的 AI Agent Platform，支持以下三种运行模式，并允许在前端以开关方式自由组合：

1. **LLM 模式**
    用户问题 → LLM 直接回答
2. **RAG 模式**
    用户问题 → 向量检索 → 相关文档注入 → LLM 回答
3. **Agent 模式**
    用户问题 → LLM 判断是否调用工具 → MCP Tool 执行 → LLM 汇总结果
4. **RAG + Agent 联合模式**
    用户问题 → RAG 检索补充上下文 → LLM 决定直接回答或调用 MCP Tool → 输出最终结果

项目核心目标不是“做一个聊天 Demo”，而是做一个**工程化、可扩展、可观测、可热插拔的 Java AI Agent Platform**。

## 2. 技术选型

### 2.1 后端

- Java 21
- Spring Boot 3.5
- LangChain4j
- MySQL
- Milvus
- Redis（可选，用于缓存/会话加速）
- SSE 作为默认流式输出方式

LangChain4j 当前对 Spring Boot 的集成要求 Java 21 和 Spring Boot 3.5；其工具箱覆盖 Prompt、Memory、Tools、RAG、Streaming、MCP 等能力，适合作为本项目唯一 AI 核心框架。

### 2.2 模型

- Chat Model：你现有的 `qwen3.5 32b-a3b`（llama server 暴露 OpenAI-compatible API）
- Embedding Model：你现有的 `bge-m3`（OpenAI-compatible API）

LangChain4j 的 OpenAI 集成可用于接入 OpenAI-compatible API，本地和云端都适配这一模式。

### 2.3 文档解析

- 首选：`ApacheTikaDocumentParser`
- 补充：PDFBox / POI / Markdown Parser 按需扩展

LangChain4j 已有 Apache Tika 文档解析集成；Tika 能自动识别并提取 PDF、DOC、PPT、XLS 等多种格式文本，适合你“尽可能多支持类型”的要求。

### 2.4 前端

- Vue3
- Naive UI / Element Plus 二选一
- 使用 SSE 展示 streaming
- 独立 Trace 面板展示 RAG 检索、Tool 调用、MCP 执行、最终答案

## 3. 核心设计原则

### 3.1 LangChain4j 作为唯一 AI 编排核心

本项目不引入 Spring AI，不做双框架混用。原因很简单：

- 你的核心目标是学习 LangChain4j
- LangChain4j 已覆盖你需要的主能力
- 双框架混用会增加抽象重叠、调试复杂度和维护成本

### 3.2 RAG 与 Agent 不互斥

RAG 负责**提供知识上下文**，Agent 负责**执行动作能力**。
 两者不是替代关系，而是“上下文增强”和“外部行动能力”的组合。

### 3.3 MCP 可热插拔

Anthropic MCP 是一个开放协议，LangChain4j 现已支持与 MCP server 通信；MCP 支持 `stdio` 与 `Streamable HTTP` 两类传输。你的平台中，MCP 不应写死在代码里，而应设计为**注册中心 + 动态启停 + 运行时装配**。

### 3.4 Memory 与 History 分离

LangChain4j 提供的是 **ChatMemory**，并明确区分了 memory 与 history：
 memory 是发给模型的“可裁剪上下文”，history 是前端展示和审计所需的“完整会话记录”。Trace 则是内部执行事件流，不应混入 history。所以本项目应采用：

- **LangChain4j ChatMemory**：用于模型上下文
- **MySQL Conversation History**：用于完整消息留存、审计、重放
- **MySQL Trace Event**：用于内部执行链路追踪与调试

这也是你“持久化一律进 MySQL”的最佳落地方式。

## 5. 模式编排设计

### 5.1 模式开关

前端请求中携带：

```
{
  "enableRag": true,
  "enableAgent": true,
  "knowledgeBaseIds": [1, 2, 5],
  "conversationId": "xxx"
}
```

配置优先级建议从现在就固定：

- **本次请求参数 > 会话配置 > 系统默认配置**
- 请求里的覆盖项只对当次生效，不自动回写会话配置
- 持久化修改统一走会话 settings 接口

### 5.2 执行逻辑

#### 模式 A：全部关闭

```
用户问题 → ChatModel → 答案
```

#### 模式 B：只开 RAG

```
用户问题 → Embedding → Milvus 检索 → 拼接上下文 → ChatModel → 答案
```

#### 模式 C：只开 Agent

```
用户问题 → ChatModel(带 Tools) → 是否调用 MCP Tool → 工具结果 → ChatModel 汇总
```

#### 模式 D：RAG + Agent 全开

```
用户问题
  → RAG 检索
  → 检索结果注入上下文
  → ChatModel(带 Tools)
  → 模型判断：
      1. 直接回答
      2. 调用 MCP Tool
  → 汇总结果
```

这里建议**RAG 直接检索，不做额外“先判断要不要检索”**。因为 LangChain4j 的 RAG API 已支持构建简单到复杂的检索管线，而先加一层 LLM 判断通常只会增加一次额外推理成本和时延。

## 6. RAG 设计

### 6.1 文档处理链路

```
文件上传
  → 文档解析（Tika）
  → 文本清洗
  → 分块
  → 向量化（bge-m3）
  → 入库 Milvus
```

LangChain4j 的 `Document` / `Metadata` / `TextSegment` 是标准 RAG 建模入口，适合在文档导入时统一附加 `knowledgeBaseId`、`fileId`、`tenantId`、`sourceType` 等元数据。

### 6.2 多知识库、多选策略

这里我给你一个明确建议：

**第一版采用：单一 collection + 手动 partition = 一个知识库一个 partition。**

原因：

- 你要求“知识库多选”
- Milvus 的 partition 允许**同一 collection 下隔离数据并支持跨多个 partition 查询**
- 这种模式非常适合“用户勾选多个知识库后联合检索”

Milvus 官方多租户说明里，partition-level 策略支持对单个或多个 partition 查询；而 collection-level 虽然隔离更强，但天然不适合“多知识库一次检索”。

#### 第一版规则

- 同一 embedding 维度、同一 schema 的知识库，共用一个 collection
- 每个知识库对应一个 partition
- 用户前端勾选多个知识库时，后端按 partition list 查询
- 该方案仅作为 **v1 学习版** 方案，不视为最终长期架构

#### 第二版升级方向

如果未来知识库数量极大，或者你要支持超大规模租户隔离，可以考虑 **partition key**；Milvus 的 partition key 策略扩展性最高，但隔离粒度和灵活性会弱一些。
后续也可以评估 metadata filter 或按 collection 拆分的方案，但前提仍然是 embedding 模型、维度、schema、距离度量保持一致。

## 7. Agent 与 MCP 设计

### 7.1 总体原则

MCP 在你的系统中不直接暴露给用户，而是：

- MCP Server 负责提供能力
- 平台负责注册/发现/启停 MCP
- LangChain4j 负责把 MCP 能力作为 Tool 暴露给模型
- 模型根据上下文自主决定是否调用

MCP 规范中，tools 本身就是“模型控制”的能力单元，这与你的设计完全一致。

### 7.2 MCP 热插拔设计

#### MySQL 表：`mcp_server`

- id
- name
- transport_type（STDIO / STREAMABLE_HTTP）
- endpoint
- command
- args_json
- headers_json
- auth_type
- auth_config_json
- enabled
- health_status
- created_at
- updated_at

#### MySQL 表：`mcp_tool_snapshot`

- id
- mcp_server_id
- runtime_tool_name（运行时全局唯一，如 `serverAlias__toolName`）
- origin_tool_name（远端 MCP 返回的原始工具名）
- tool_title
- tool_description
- input_schema_json
- enabled
- version
- synced_at

### 7.3 运行机制

1. 前端新增/编辑 MCP server
2. 后端建立连接
3. 拉取工具清单并缓存
4. 注册到运行时 Tool Registry
5. 新请求到来时，为当前会话动态组装可用工具集
6. 模型决定是否调用工具

### 7.4 传输策略建议

- **本地开发工具**：优先 `stdio`
- **远程服务型工具**：优先 `Streamable HTTP`

LangChain4j 官方 MCP 文档明确说明同时支持这两种传输。

## 8. Memory 设计

### 8.1 为什么要双层存储

LangChain4j 的 ChatMemory 适合做“发给模型的上下文管理”，但官方明确说它不是完整 history。

所以必须拆成两层：

#### A. 会话历史（History）

存 MySQL，全量存储：

- user message
- assistant message
- tool call
- tool result
- rag snippets
- trace event

#### B. 模型记忆（Memory）

使用 LangChain4j：

- `MessageWindowChatMemory` 或 `TokenWindowChatMemory`
- 自定义 `ChatMemoryStore`，底层落 MySQL

### 8.2 具体建议

第一版先用：

- `TokenWindowChatMemory`
- 自定义 `ChatMemoryStore(MySQL)`

原因是 token window 更适合真实场景，能比 message count 更稳地控制上下文长度与成本。LangChain4j 当前内置的开箱实现就是 `MessageWindowChatMemory` 与 `TokenWindowChatMemory`。

------

### 8.3 并发控制建议

Memory 绑定在 conversation 维度后，同一个会话不能同时跑多个生成任务，否则容易出现 memory 视图错乱。第一版建议直接采用：

- **conversation 维度串行执行**
- 同一会话已有请求执行中时，新请求要么排队，要么直接返回“稍后重试”
- `requestId` 作为一次生成任务的唯一标识，History、Trace、SSE 都围绕 `requestId` 关联

## 9. Streaming 与 Trace 设计

### 9.1 Streaming

LangChain4j 提供 `StreamingChatModel` 与响应处理器，适合后端通过 SSE 向前端持续推送 token。

为了降低前端复杂度并统一 Trace 订阅方式，第一版建议采用 **两步式 + 双 SSE 通道**：

1. `POST /api/v1/chat/send`：创建一次聊天请求，返回 `requestId`
2. `GET /api/v1/chat/stream/{requestId}`：订阅回答 token 流
3. `GET /api/v1/traces/stream/{requestId}`：订阅 Trace 事件流

说明：

- `POST + text/event-stream` 也可以实现流式，但前端通常需要 `fetch` 流式读取或自定义 SSE 客户端
- 如果希望前端直接使用浏览器原生 `EventSource`，则更适合使用 `GET + requestId` 的订阅方式
- 聊天流和 Trace 流分开后，聊天区与 Trace 面板能自然解耦

### 9.2 Trace 事件模型

建议统一事件总线，前后端共用以下事件类型：

- `USER_MESSAGE_RECEIVED`
- `MEMORY_LOADED`
- `RAG_RETRIEVAL_STARTED`
- `RAG_RETRIEVAL_FINISHED`
- `RAG_SEGMENTS_SELECTED`
- `MODEL_DECISION_STARTED`
- `TOOL_CALL_REQUESTED`
- `TOOL_CALL_FINISHED`
- `FINAL_RESPONSE_STREAMING`
- `FINAL_RESPONSE_COMPLETED`
- `ERROR_OCCURRED`

落地时建议遵循两条规则：

- `conversation_message` 只保存可重放的消息内容：`USER` / `ASSISTANT` / `TOOL_CALL` / `TOOL_RESULT` / `RAG_CONTEXT`
- `trace_event` 保存内部过程事件，优先映射 LangChain4j 原生 observability/listener 事件，平台特有事件再补自定义埋点

### 9.3 前端展示

前端至少展示三栏：

1. **聊天区**：最终答案流式输出
2. **Trace 区**：时间线显示检索、工具、模型决策
3. **上下文区**：显示检索到的文档片段 / 工具入参与返回摘要

## 13. 关键实现建议

### 13.1 第一版不要优先使用 `langchain4j-agentic`

因为官方仍将 agentic 模块视为 experimental。你的第一版更适合走稳定路线：

**AI Services + Tools + RAG + MCP + Memory**

这样项目更稳、学习路径更清晰。

### 13.2 MCP 不要直接“全量暴露”

要有：

- 启用/禁用
- 白名单
- 每个会话可见工具列表
- 人工确认开关（高风险工具）
- 远程 MCP server 的 endpoint 不对普通用户裸开放
- `STREAMABLE_HTTP` 场景明确 Origin 校验与认证要求

MCP 规范也强调应保留 human-in-the-loop 的否决能力。

### 13.3 RAG 统一 embedding 维度

同一 collection 下的知识库必须使用相同 embedding 模型与维度；如果未来要混用 embedding 模型，拆 collection，不要强塞进一个 collection。

### 13.4 文档解析先保守支持

第一版建议只保证：

- pdf
- docx
- md
- txt

然后再逐步打开 ppt/xlsx/html 等；虽然 Tika 支持很多格式，但解析质量仍应逐项回归测试。

## 14. 最终结论

这个项目方向是对的，而且**已经不是“练手小项目”了，而是一个可以放到 GitHub 作为核心代表作的 Java AI Agent Platform**。

最终技术路线我建议定为：

- **核心框架**：LangChain4j
- **后端**：Spring Boot 3.5 + Java 21
- **模型**：Qwen（OpenAI-compatible）
- **Embedding**：bge-m3（OpenAI-compatible）
- **向量库**：Milvus
- **文档解析**：Apache Tika
- **协议**：Anthropic MCP
- **会话**：MySQL 持久化 + LangChain4j ChatMemory
- **前端**：Vue3 + SSE + Trace 面板

如果按这个路线推进，你能同时学到：

- LangChain4j 的主干能力
- Java 侧 RAG 的工程化实现
- Agent + Tool Calling 的真实落地
- MCP 的平台化接入方式
- AI 应用的前后端协同与可观测设计
