# Open Agent Platform

## 模块拆分与目录结构设计文档

## 1. 版本基线

### 1.1 推荐基线

- JDK 21
- Spring Boot 3.5.11
- LangChain4j + Spring Boot Starter
- MyBatis
- MySQL 8.x
- Milvus 2.4.x（你现有环境可先沿用）
- Vue3 前端
- SSE 流式输出

说明：

- 截至 2026-03-11，LangChain4j 官方 Spring Boot 集成文档仍以 Java 17+ 和 Spring Boot 3.5 为主。
- 当前仓库已统一切换到 `Spring Boot 3.5.11`，与 LangChain4j 官方 Spring Boot 集成口径保持一致。

Milvus Java SDK 与 MyBatis 对 Java 版本要求相对宽松，因此这里的关键约束主要来自 LangChain4j 与 Spring Boot 3.5 的集成方式和版本配套。

## 5. 核心接口设计

## 5.1 会话编排核心接口

```
public interface ChatOrchestrator {
    void chat(ChatRequest request, ChatStreamObserver observer);
}
```

职责：

- 读取前端开关
- 装载 memory
- 决定是否启用 RAG
- 决定是否启用 Agent
- 组装 tool 集
- 推送 trace
- 推送最终 answer

### 5.2 RAG 检索接口

```
public interface KnowledgeRetrievalService {
    RetrievalResult retrieve(String query, List<Long> knowledgeBaseIds);
}
```

### 5.3 文档入库接口

```
public interface KnowledgeIndexService {
    void indexFile(Long knowledgeBaseId, UploadedFile file);
}
```

### 5.4 MCP 注册接口

```
public interface McpServerRegistryService {
    Long register(McpServerCreateCommand command);
    void enable(Long serverId);
    void disable(Long serverId);
    void refreshTools(Long serverId);
    List<McpToolSnapshot> listAvailableTools();
}
```

### 5.5 Tool 装载接口

```
public interface ToolAssemblyService {
    List<Object> assembleTools(ChatContext context);
}
```

这里用 `Object` 是因为 LangChain4j 的工具暴露方式可以是注解方法或适配对象，真正实现时你可以再收紧成统一包装。LangChain4j 的工具调用主线依然是围绕 tool/function calling 展开的。

### 5.6 Memory 接口

```
public interface ConversationMemoryService {
    ChatMemory getOrCreateMemory(Long conversationId);
    void persistHistory(Long conversationId, MessageRecord record);
}
```

实现约束：

- `conversationId` 同时也是 memory 串行化的基本单位
- 不允许同一会话并发更新 ChatMemory

## 6. 核心执行流程细化

## 6.1 聊天主流程

```
1. 前端发送 ChatRequest
2. 生成 requestId
3. 按 conversationId 获取执行锁（同会话串行）
4. 加载 conversation 持久配置
5. 合并本次请求覆盖项，得到 effective settings
6. 加载 ChatMemory
7. 如果 enableRag=true：
      进行知识库检索
      写 trace
8. 如果 enableAgent=true：
      装载 MCP tool 列表
      生成 runtimeToolName（全局唯一）
      写 trace
9. 组装 prompt + memory + rag context + tools
10. 调用 StreamingChatModel
11. 过程中持续输出：
      token
      trace
      tool call
      tool result
12. token 流推送到 /api/v1/chat/stream/{requestId}
13. trace 流推送到 /api/v1/traces/stream/{requestId}
14. 持久化 assistant answer
15. 持久化完整 trace
16. 释放会话执行锁
```

LangChain4j 提供 StreamingChatModel 支持流式响应；ChatMemory 负责给模型传递上下文而不是保存完整历史。

## 7. RAG 模块细化设计

## 7.1 文档处理流水线

```
上传文件
→ 文件元数据入库
→ Tika 解析
→ 规范化清洗
→ 文本分块
→ embedding
→ 写入 Milvus
→ segment 元数据入 MySQL
```

LangChain4j 已提供 Apache Tika 文档解析集成；Milvus 集成也已有官方 embedding store。

## 7.2 分块策略建议

第一版建议：

- chunk size：800～1200 字符
- overlap：100～200 字符

分块策略不要一开始做太复杂，先保证：

- 检索到的片段能读懂
- 上下文不碎裂
- 单次 prompt 不爆 token

演进建议：

- v1 先使用字符维度固定分块，优先跑通索引链路
- v1.1 / 第二阶段再切到 recursive token splitter，与 `TokenWindowChatMemory` 的口径逐步统一

## 7.3 多知识库检索

建议规则：

- 同 embedding 模型
- 同维度
- 同 collection
- 每个知识库一个 partition
- 多选时按 partition 列表检索

这样最符合你“知识库多选”的产品设计，也最方便前端实现“勾选多个知识库一起问”。

## 8. Agent 模块细化设计

## 8.1 Agent 不是单独的大黑盒

第一版不要做“一个巨型 Agent 内部自己处理所有事”，而要做成：

- Orchestrator 负责流程选择
- ToolAssemblyService 负责装配工具
- ChatModel 负责做出工具调用决策
- MCP 模块负责提供工具能力

也就是说：

```
平台负责编排
模型负责判断
MCP 负责执行
```

## 8.2 Tool 分类

建议把工具分三类：

### A. 系统工具

由平台自带：

- 当前时间
- 会话信息
- 选择的知识库信息

### B. MCP 工具

来自外部 MCP server：

- 文件操作
- 网络检索
- 命令执行
- 第三方系统查询

### C. 平台内工具

后续可自行开发：

- 查询平台数据库
- 查询知识库清单
- 查询 trace
- 查询历史会话

## 9. MCP 模块细化设计

## 9.1 MCP 连接模型

MCP server 配置建议支持两类：

### `STDIO`

适合本机命令型 MCP server

字段：

- command
- args_json
- env_json

### `STREAMABLE_HTTP`

适合远程 MCP 服务

字段：

- endpoint
- headers_json
- auth_config_json

LangChain4j 官方文档明确写到支持 `stdio` 与 `Streamable HTTP`。

## 9.2 热插拔实现逻辑

```
前端新增 MCP
→ 保存到 MySQL
→ 后端建立连接
→ 获取 tool 列表
→ 保存 snapshot
→ 标记 server enabled
→ 新会话请求时自动装配
```

## 9.3 为什么要做 snapshot

因为你前端需要展示：

- 这个 MCP 有哪些工具
- 工具说明是什么
- 当前是否启用
- 上次同步时间

所以工具清单要有本地快照，而不是每次都实时去拉。

## 10. Memory 模块细化设计

## 10.1 推荐方案

第一版：

- `TokenWindowChatMemory`
- 自定义 `ChatMemoryStore`
- 底层 MySQL

LangChain4j 内置了窗口型 memory 方案，你只需要把存储落地到 MySQL。

## 10.2 MySQL 持久化拆分

建议分两张表：

### `chat_memory_message`

保存“送给模型的上下文消息”

### `conversation_message`

保存“完整历史消息”

这样你后续做：

- 历史重放
- 前端完整展示
- 重新构建 memory
   都会更灵活。

## 11. Trace 模块细化设计

## 11.1 Trace 事件结构

```
public class TraceEvent {
    private String eventType;
    private String eventSource;
    private String requestId;
    private Long conversationId;
    private Long messageId;
    private Long timestamp;
    private Object payload;
}
```

## 11.2 第一版事件类型

- USER_MESSAGE_RECEIVED
- MEMORY_LOADED
- RAG_STARTED
- RAG_FINISHED
- RAG_SEGMENTS_SELECTED
- TOOLS_ASSEMBLED
- TOOL_CALL_STARTED
- TOOL_CALL_FINISHED
- MODEL_STREAMING
- ANSWER_COMPLETED
- ERROR_OCCURRED

事件来源建议：

- `LANGCHAIN4J_NATIVE`：LangChain4j 原生 observability / listener 事件
- `APP_CUSTOM`：平台自定义事件，例如会话锁竞争、人工确认结果、工具运行时命名结果

## 11.3 前端展示方式

建议不用一开始就追求 LangSmith 那种复杂 DAG，可先做：

- 左侧：聊天内容
- 右侧：trace timeline
- 下方抽屉：检索片段 + tool 出入参

这样你前端工作量不会爆炸，但已经足够体现“平台感”。
