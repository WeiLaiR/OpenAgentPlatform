# 当前实现清单

更新日期：2026-04-24

## 文档定位

这份文档只记录**当前已经落地并进入主链路的能力结论**。

它不再逐条铺开所有接口和页面细节；更细的 DTO、VO、接口列表、数据库设计和 Prompt 说明，请分别查看 `docs/` 下的专题文档。

## 当前阶段结论

当前项目已经进入“**有状态会话版的第一版平台基线**”阶段，核心特征如下：

- 四种模式 `LLM / RAG / Agent / RAG + Agent` 已能在统一主链路中切换
- 聊天、知识库、MCP、Trace 已形成前后台协作闭环
- LangChain4j 官方主线能力已在主链路上落地到 `ChatModel / StreamingChatModel / AI Services / ChatMemory / RAG / MCP / Trace`
- 当前剩余主缺口已基本收敛到 `待确认工具调用的状态回放/异常治理` 和 `更完整 observability`

## 已落地的核心能力

### 1. 基础聊天与会话主链路

- 已具备统一响应、统一错误模型、全局异常处理和 `X-Request-Id` 透传。
- 已具备同步聊天与两步式流式聊天能力，流式结果和 Trace 都支持 SSE 订阅。
- 已具备会话创建、会话列表、历史消息查询、会话配置读取与更新、会话级 memory 清理。
- 已实现“空草稿会话”工作方式：进入聊天页或点击“新聊天”时不预创建空会话，首轮真正发送后才落库。
- 已实现新会话首轮回答后的 AI 自动命名，且仅在首次生效。

### 2. 模式编排与执行收口

- 已形成统一编排骨架：`ChatOrchestrator + ModeResolver + ChatExecutionSpec + ChatOutputPort`。
- 已把输出、RAG runtime、Tool runtime、平台 Trace 事件从编排主类中拆出到 `BufferedChatOutputPort / StreamingChatOutputPort / RagRuntimeResolver / ToolRuntimeResolver / TracePublisher`。
- 已将旧的 `ChatGenerationService` 收敛为兼容层，真正主入口切到 `ChatOrchestrator`。
- 已实现“请求覆盖 > 会话配置 > 系统默认”的统一解析，RAG 与 MCP runtime 都消费统一的会话级有效配置。
- 已具备会话维度最小串行保护，避免同一会话并发生成破坏主链路状态。

### 3. Prompt 主链路

- 已在代码中固定第一版最小 Prompt 集合：`system-base / rag-context / agent-tool-policy / mcp-tool-selection / answer-format`。
- 已由 `ChatContextAssembler` 统一承担非 Agent 路径的 Prompt 装配，正式形成 `system -> history -> user` 的请求结构。
- 已把 RAG Prompt 收口进统一 Prompt 目录，继续复用 LangChain4j 官方 `PromptTemplate + DefaultContentInjector`。
- 已把 Agent 路径的 system prompt 注入到官方 `AI Services` 主链路，由 `AgentAiServiceFactory` 统一装配。
- 已具备 `PromptAssembly` 摘要与调试视图，不默认暴露完整 Prompt 原文。

### 4. Memory 与会话上下文

- 已正式接入 LangChain4j 官方 `ChatMemory` 主线能力，优先使用 `TokenWindowChatMemory`。
- 当当前模型无法被官方 tokenizer 识别时，已自动降级到官方 `MessageWindowChatMemory`，避免非 OpenAI 模型启动失败。
- 已支持通过 `openagent.ai.chat.tokenizer-model-name` 为 OpenAI-compatible 非 OpenAI 模型显式指定 tokenizer 参考模型名。
- 已实现 `ChatMemoryStore(MySQL)`，把 memory 视图持久化到 `chat_memory_session / chat_memory_message`。
- 已实现 `conversationId -> memoryId` 的稳定映射。
- 已实现 history 与 memory 分离：完整历史保存在 `conversation_message`，模型上下文窗口保存在 `chat_memory_message`。
- 已实现“请求成功后才回写 ChatMemory”，避免失败轮次污染上下文。

### 5. 知识库与 RAG

- 已具备知识库创建、列表、文件上传、文件列表、文件删除、文件索引、检索测试、片段编辑、单片向量重算等基础管理闭环。
- 已实现 LOCAL 文件落盘、`file_hash / storage_uri` 入库以及知识库级 `kb_{knowledgeBaseId}` Milvus partition 管理。
- 已具备基于 Tika 的 `txt / md / docx` 文档解析。
- 已具备段落优先的基础分片、分片向量生成、Milvus 写入与失败状态回写。
- 已实现按知识库分区检索 Milvus 并回查 MySQL 原文片段。
- 已将聊天主链路中的 RAG 回切到 LangChain4j 官方 `DefaultRetrievalAugmentor + EmbeddingStoreContentRetriever`。
- 已通过 `KnowledgeTextSegmentEmbeddingStoreFactory` 把现有检索面适配为 LangChain4j `EmbeddingStore<TextSegment>`。
- 已确保 retrieved content 只进入本轮请求上下文，不回写 `ChatMemory`。

### 6. MCP 与 Agent

- 已具备 MCP Server 的新增、编辑、连通性测试、工具同步与工具启停管理。
- 已通过 LangChain4j 官方 MCP client 完成 MCP Server 连接与工具发现。
- 已在同步阶段处理远端工具 description 过长的问题，避免直接导致快照入库失败。
- 已把 Agent / RAG + Agent 主链路切到 LangChain4j 官方 `AI Services + ToolProvider + TokenStream`。
- 已支持基于会话默认配置与请求级覆盖动态装配 MCP Tool。
- 已在运行时接入统一工具风险判定：`LOW / MEDIUM -> ALLOW`、`HIGH -> REQUIRE_CONFIRMATION`。
- 已新增 `agent_tool_confirmation` 持久化表，用于保存高风险工具的待确认记录、原始 `tool request`、续跑 `requestId` 与决策状态。
- 已提供 `POST /api/v1/chat/tool-confirmations/{confirmationId}/approve|reject`，并在确认/拒绝后继续完成同一轮工具调用与模型回答。
- 已支持同步返回与流式 `message_end` 返回 `tool_confirmation_required + pendingConfirmation` 结构化状态，前端聊天页可直接在当前 assistant 气泡上执行确认/拒绝。
- 已提供会话维度 pending confirmation 查询能力，页面刷新或重新进入会话后可恢复待确认工具调用，不再只依赖前端内存状态。
- 已在 pending confirmation 查询时处理已过期记录，超时未处理的 `PENDING` 会推进为 `EXPIRED` 并从可操作列表中剔除。
- 已在 continuation 路径中固定复用已记录的 `tool request`，不会重新让模型生成一次工具调用。

### 7. Trace 与 observability

- 已具备 `GET /api/v1/traces/{requestId}` 与 `GET /api/v1/traces/stream/{requestId}` 两类 Trace 能力。
- 已形成 `APP_CUSTOM + LANGCHAIN4J_NATIVE` 双层事件结构。
- 已通过 `ChatModelListener + LangChain4jNativeTracePublisher + ChatModelObservationContextHolder` 接入模型原生事件。
- 已接入 `PROMPT_ASSEMBLY_RESOLVED`、`CHAT_EXECUTION_SPEC_RESOLVED`、`RAG_RETRIEVAL_*` 等关键平台事件。
- 当前 Trace 已能覆盖模型请求、响应、错误，以及主链路中的 Prompt、配置、RAG 等关键节点。
- 已补充 `TOOL_RISK_EVALUATED`、`TOOL_CONFIRMATION_REQUIRED`、`TOOL_CONFIRMATION_APPROVED`、`TOOL_CONFIRMATION_REJECTED`、`TOOL_CONFIRMATION_CONTINUATION_STARTED`、`TOOL_EXECUTION_BLOCKED` 等 Agent 风控事件。

### 8. 前端信息架构与交互

- 已完成前台聊天区与后台管理区的路由分层：默认入口为 `/chat`，后台收口到 `/admin/system`、`/admin/knowledge`、`/admin/mcp`。
- 已完成聊天页从“三栏工作台”收敛到“侧边栏 + 对话主区”的第一版结构。
- 已完成会话列表、会话排序、会话快捷操作、输入区回焦、消息自动跟随滚动等核心交互补强。
- 已实现聊天页会话级默认配置与请求级 `KB / MCP` 覆盖语义的前端收口。
- 已为 `ASSISTANT` 消息补齐受控 Markdown 渲染和安全清洗。
- 已把 Trace 从长期常驻面板收敛为消息侧摘要入口 + 弹窗详情，降低主聊天区干扰。
- 已具备知识库管理页、MCP 管理页、系统页和相应的前后端联调入口。

### 9. 测试与脚本基线

- `backend/scripts/` 已形成固定脚本入口：环境检查、测试、启动、构建、通用 Maven Wrapper 入口。
- `backend/sql/ddl/` 已形成统一 DDL 维护目录与 README 约定。
- 后端测试目录已按 `ai / chat / conversation / knowledge / mcp / memory / trace` 分组维护，主链路核心模块都有对应测试类。

## 当前与目标基线的主要差距

当前真正还没有收口的主缺口已经比较集中：

- 待确认工具调用已具备刷新恢复与基础过期治理，但确认/拒绝/执行后的状态回放、失败展示和异常中断治理还可以继续补强
- LangChain4j observability 仍停留在“基础可用”，还未扩展到更完整的 AI Service / Tool / 风控决策生命周期
