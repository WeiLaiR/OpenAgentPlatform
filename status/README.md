# status 目录说明

`status/` 用来维护**当前实现状态**，和 `docs/` 中的设计说明分开。

这里不负责展开设计细节，只回答两个问题：

- 现在已经实现了什么
- 现在最应该先做什么

## 当前文件职责

- `当前实现清单.md`
  - 记录已经落地并进入主链路的能力结论
  - 不写计划项，不逐条铺所有接口细节
- `当前优先实现清单.md`
  - 记录当前阶段最优先补的事项
  - 不写长期路线图，不堆愿望清单

## 维护顺序

每次完成一个功能批次后，按下面顺序维护：

1. 先更新 `当前实现清单.md`
2. 再对照 `docs/` 判断目标基线是否变化
3. 最后更新 `当前优先实现清单.md`

## 与 docs 的分工

- `status/`：写当前状态和当前优先级
- `docs/`：写设计原因、目标基线和专题说明

如果同一个内容既要写“现状”，又要写“为什么这样设计”，优先：

- 把现状写进 `status/`
- 把设计意图写进 `docs/`
